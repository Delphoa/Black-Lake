## Research Notes

### Problem

Signal classes produced by translations, scaling, warping, or other mass movement can be nonlinear in raw coordinates. The paper asks when CDT, R-CDT, or LOT coordinates make an algebraically generated class convex, so a simpler linear classifier or estimator may become appropriate.

### Mechanism

Signals are compactly supported, nonnegative, unit-mass functions. A class is generated by applying differentiable invertible spatial transformations to a template with a Jacobian mass correction. Each signal is represented by the quadratic-cost optimal transport map from a fixed reference.

### Main Results

1. In 1D, Theorem 3.2 gives an exact criterion: the transformed class is convex for every template if and only if the inverse transformation set is convex. For a group, this becomes convexity of the group.
2. The paper gives translation, positive-scaling, increasing-affine, and fixed-point examples and proves there are uncountably many distinct convex subgroups.
3. In `d >= 2`, requiring the composition property for every signal limits the admissible maps to translations and isotropic scaling. A restricted 2D signal family permits a larger group with diagonal stretching.

### Evidence and Limits

The only new in-paper experiment visualizes two synthetic classes with 500 randomly generated fifth-degree polynomial deformations per class. LDA projections overlap in signal space and separate after CDT. The source supplies a compelling illustration but no accuracy, uncertainty, repeated seeds, geometry metrics, robustness sweep, or official reproduction package. Practical claims therefore remain conditional on the algebraic generative model, transform validity, class disjointness, reference choice, numerical solver, and finite-sample behavior.

### Implementation Relevance

The useful engineering pattern is a gate, not a shortcut: declare the transformation family; validate signal mass/support and map regularity; compute the transform with diagnostics; measure margin and convex-hull violations; compare matched native/transformed baselines; and generate off-model counterexamples. A transform should not be assumed to simplify arbitrary real data merely because it derives from optimal transport.

## Synthesis Note

### Concept Bridge

Across the selected paper and the three related entries, representation design is treated as a contract between data generation, geometry, and the downstream operator. Transport Convexity formalizes the contract: when a deformation family is convex in the relevant inverse/composition sense, the transformed class inherits convexity. Physical Data AI encodes a similar contract through a compact physical equation; GPMD encodes it in the optimization geometry; Learn to Pool learns it from compression-aware examples. The shared lesson is to specify why a representation should make an operation simple and then measure both the promised simplification and the boundary where it fails.

### Potential Implementations

1. Build a local transform-geometry audit that compares native and CDT feature spaces using held-out linear accuracy, margins, hull violations, reconstruction, solver diagnostics, and assumption-failure sweeps.
2. Build a transport-aware delay/dispersion estimator that fits a bounded linear subspace in CDT space and rejects signals that fail positivity, mass, monotonicity, residual, or extrapolation gates.
3. Build a representation-contract registry that records a model's assumed transformation family, allowed input domain, reference choice, simplification metric, counterexample generator, and deployment stop conditions.

### Deeper Relationship Observations

1. Convexity is useful because it moves complexity rather than erasing it: transport solving, physical simulation, mirror-map updates, or pooling-aware training pays an upstream cost so a downstream operator can be simpler.
2. Each related artifact has an assumption-mismatch failure mode: non-transport signals, non-PDE-aligned data, non-tabular/nonconvex control, or unseen pooling behavior. Geometry claims are therefore incomplete without a domain-fit detector.
3. The selected paper's 1D-versus-multidimensional split mirrors a broader systems pattern: properties that look compositional in a restricted coordinate system may collapse when the representation family is expanded, so scope boundaries must be part of the interface.

### Conceptual Similarities

1. All four artifacts intentionally reshape or choose a representation so a simpler downstream computation becomes effective.
2. All four rely on explicit structure—transport groups, physical equations, convex regularizers, or pooling exposure—rather than unconstrained feature learning alone.
3. All four require matched evaluation: linear separation, classification accuracy, convergence/error floors, or retrieval retention must be measured under the exact representation operation being claimed.

### MVP Implementations

1. **Density validity gate.** Refuse implicit normalization and record why a signal is ineligible.

```python
import numpy as np

def density_status(x: np.ndarray, y: np.ndarray) -> str:
    if x.ndim != 1 or y.shape != x.shape or len(x) < 8:
        return "invalid-shape"
    if np.any(~np.isfinite(y)) or np.any(y < 0):
        return "invalid-values"
    mass = np.trapezoid(y, x)
    return "ok" if abs(mass - 1.0) <= 1e-3 else "invalid-mass"
```

2. **Held-out margin evidence.** Compare the same linear-probe family in native and transformed features.

```python
import numpy as np

def signed_margins(features: np.ndarray, labels: np.ndarray,
                   weight: np.ndarray, bias: float) -> np.ndarray:
    if set(np.unique(labels)) - {-1, 1}:
        raise ValueError("labels must be -1 or 1")
    return labels * (features @ weight + bias)
```
