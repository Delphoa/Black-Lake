# Meta-Latent Registers as a Shared Interface for Generation, Retrieval, and Compression

## A detailed review, technical reconstruction, and independent re-conceptualization of “GRC: Unifying Reasoning-Driven Generation, Retrieval and Compression”

**Paper:** Zhongtao Miao, Qiyu Wu, and Yoshimasa Tsuruoka, *GRC: Unifying Reasoning-Driven Generation, Retrieval and Compression*, arXiv:2605.09100v2, May 2026.[^1]  
**Canonical record:** https://arxiv.org/abs/2605.09100  
**Reviewed evidence:** locally archived full PDF and full-paper HTML for the current revision; canonical arXiv record; paper-linked code URL; BRIGHT primary sources; and GritLM primary sources.[^2]  
**Coverage:** complete main paper, training/evaluation appendices, six numbered equations, fifteen tables, eight figures, prompts, latency details, author contributions, and revision notes.  
**Reproduction status:** no experiment was rerun; the paper-linked GRC code URL was unavailable at inspection.

---

## Executive assessment

### The paper in one sentence

GRC trains one causal decoder so that learned meta-latent register tokens can simultaneously support answer generation, reasoning-conditioned embeddings, and fixed-length context KV caches, then serves those heterogeneous caches with a paged-attention variant.

### The deeper idea

GRC treats generation, retrieval representation, and compression as three consumers of one internal state rather than three separate applications. A sequence contains ordinary user/assistant tokens, 128 learned register embeddings, and a reconstruction segment. The registers attend to the original context. The reconstruction segment is prevented from reading the original context and must reconstruct it through the registers. The final register hidden states are adapted, mean-pooled, and normalized into an embedding. Ordinary answer tokens are trained with causal language modeling. A weighted sum of generation, reconstruction, and contrastive losses therefore pressures the same register bank to become both a compressed generative memory and a retrieval vector source.

At inference, the architecture supports four modes: ordinary generation; reasoning followed by latent query embedding; document embedding; and generation conditioned on a document’s stored register KV cache. The serving contribution, hybrid paged attention (HPA), adds compressed register-cache blocks to a paged KV manager alongside ordinary prefix and dynamic caches.

### Bottom-line judgment

The paper offers an ambitious and coherent interface design. Its strongest scientific evidence is breadth: the 4B model reports an average 27.2 nDCG@10 on BRIGHT, 38.06 sacreBLEU and 66.63 chrF on PwC reconstruction, 67.40/64.27 on GSM8K/BBH, and 45.8% on the sampled Natural Questions RAG evaluation with plain text. Compressed latent documents also improve NQ over no-document generation—40.4 versus 35.4 for 4B—showing that the fixed register cache contains usable information. The detailed HPA table reports roughly order-of-magnitude latency reductions relative to the authors’ batch-one Hugging Face-style naive implementation.

The evidence does not establish a free three-way unification. Reasoning-enhanced query embeddings generate up to 4,096 tokens before pooling in the reported RAG setup, which can be far more expensive than conventional embedding. Plain-text documents outperform compressed caches by 5.4 points for GRC-4B and 4.6 for 1.7B. Reconstruction deteriorates with document length. The fixed-length claim is conditional on a fixed register and model configuration.[^6] Only a fraction of the sampled training steps is executed, and the public code URL embedded in the paper returned 404 during vetting.[^3] No dedicated limitations section is provided, although the conclusion acknowledges a conflict when the last latent state also predicts the next token.

### Principal strengths

- A single causal mask and register interface make all three tasks composable in one model execution graph.
- Reconstruction is causally constrained: the decoder-side reconstruction segment cannot bypass the latent bottleneck.
- Retrieval, generation, compression, RAG, and serving latency are all evaluated rather than inferred from one proxy.
- The fixed number of latent tokens gives a clear KV-storage bound with respect to original document length.
- Appendices disclose training hardware, package versions, loss machinery, prompts, device allocation, and detailed latency cells.
- Negative tradeoffs are visible: compressed memory loses information, long reconstruction declines, and the final-token objective can conflict.

### Principal qualifications

- “One forward pass” applies to the configured combined sequence, but self-reason embeddings still autoregressively generate a potentially long reasoning trace.
- Fixed-length compressed KV is `O(1)` in source length only for fixed register count/model/batch/precision.
- BRIGHT gains are not compared at matched reasoning-token compute, training data, or model family.
- NQ uses 500 randomly selected test examples; the random seed and uncertainty interval are not reported.
- HPA is compared with a deliberately naive batch-one path, so the speedup does not isolate the new attention kernel against mature optimized serving systems.
- Training consumes 32 GH200s for 40 hours per model—about 1,280 GPU-hours each—despite running only a subset of the planned steps.

## 1. Problem and contribution boundary

### 1.1 Fragmented LLM application stacks

Typical RAG and long-context systems deploy an embedder, generator, and compressor as separate models. The same document may be encoded for retrieval, tokenized again for generation, and processed again for compression. Hidden states and KV caches cannot usually be reused across independently trained modules. Reasoning models amplify the cost because they add long intermediate traces.

GritLM previously unifies embedding and generation, but uses bidirectional attention for embeddings and causal attention for generation. GRC argues that heterogeneous masks obstruct simultaneous execution and leave compression outside the model. Its response is to retain causal attention and introduce continuous register embeddings that bridge all three objectives.

### 1.2 Actual novelty

The work inherits causal language modeling, contrastive retrieval learning, latent/gist compression, GradCache, paged attention, Qwen3 bases, and standard benchmarks. Its novelty lies in the specific register-mediated multitask construction; the custom attention mask that forces reconstruction through register KV; the self-reason–latent embedding sequence; the latent-memory RAG mode; and HPA’s block management for regular plus compressed caches.

### 1.3 Scope that should not be overstated

GRC is not a universal NLP model in the evidence provided. It covers selected reasoning, retrieval, reconstruction, RAG, and latency tasks with 1.7B and 4B bases. It does not evaluate general MTEB, multilingual retrieval, safety alignment, continual updates, memory deletion, or multi-document latent composition. “Three times data utilization” means training examples contribute three loss types, not that they provide three statistically independent labels or a measured threefold sample-efficiency gain.

## 2. Unified sequence and attention design

### 2.1 Three segments and register tokens (Equation 1)

The training sequence has three regions. Segment 1 is user instruction, query/document, and possibly reasoning/answer tokens. Segment 2 contains `m` learned register vectors `r_i` that are continuous parameters rather than vocabulary tokens. Segment 3 contains a reconstruction prompt and the target context. The paper uses 128 registers for both model sizes.

Because the registers are not token IDs, the model cannot emit them as gibberish text, and each register has its own learned input vector. Calling them “registers” is a useful analogy: they are a fixed-capacity scratch space into which attention writes context information. The analogy should not imply discrete address semantics; every layer produces distributed key/value and hidden representations.

### 2.2 Data transformation

Generative examples `(u,x,y)` are extended with registers, a randomly selected reconstruction prompt, and a copy of the original conversation as reconstruction target. For retrieval data, a reasoning trace is generated for each query; query, positive document, and negative documents each receive registers and reconstruction targets. Generative examples are also reused for representation learning by treating an example as its own positive and other minibatch examples as negatives.

This reuse is efficient but creates a potential shortcut. A self-positive contains nearly identical content and formatting, so its contrastive signal may be easier than semantic relevance between different query/document forms. The BRIGHT result indicates the model learns more than pure identity, but an ablation separating generative self-positives from supervised retrieval examples is needed.

### 2.3 Customized causal mask

Registers can attend to Segment 1, allowing their per-layer KV states to absorb the source. Segment 3 is masked from Segment 1 and can attend only through the register bottleneck plus its own causal prefix. Therefore successful reconstruction requires information encoded in register KV. This directionality is the main technical safeguard against simply copying the original context.

The paper says all tasks use a causal mask, but it is a customized causal/block mask rather than ordinary lower-triangular attention. The reconstruction portion has an explicit blocked region. Exact mask construction, position IDs, and cache reuse therefore matter to reproduction.

## 3. Losses and representation reconstruction

### 3.1 Generation loss (Equation 2)

`L_Gen` is average autoregressive cross-entropy over assistant response tokens in Segment 1. It also applies to document examples whose model response is the literal “None.” The model parameters and language head are trained jointly. Masking register tokens out of this loss prevents continuous registers from being treated as target vocabulary symbols.

### 3.2 Reconstruction loss (Equation 3)

`L_Recons` is cross-entropy over context tokens in Segment 3, conditioned on the reconstruction prompt, earlier reconstruction tokens, and register states, while direct Segment-1 attention is blocked. The denominator averages over reconstructed tokens. This loss trains a lossy autoencoder inside a decoder-only transformer: context-to-register compression and register-to-text reconstruction happen in one forward/backward graph.

Reconstruction fidelity is not equivalent to downstream utility. Exact lexical metrics reward surface recovery; RAG may need only answer-bearing facts. Conversely, good average reconstruction can hide catastrophic loss of rare names or numbers. The paper reports both reconstruction and NQ usefulness, which is better than relying on either alone.

### 3.3 Representation loss (Equation 4)

The last hidden states at all registers pass through adapter `A`, are mean-pooled, and normalized. A contrastive loss rewards query–positive cosine similarity over positives and negatives. In the v2 record, the authors note that typos in Equation 4 were fixed, so the current expression—not an older cached rendering—should govern implementation.[^1]

The paper writes `exp(τ·σ)` while Table 5 reports a temperature of 0.02. Many InfoNCE formulations divide similarity by temperature. If multiplication is literal, `τ=0.02` would flatten logits rather than sharpen them. The revision note says Equation 4 was corrected, but the implementation is unavailable, leaving this convention a high-priority audit item. Reproduction must determine whether the configured value is an inverse temperature or divisor.

### 3.4 Total loss (Equation 5)

The objective is `α L_Gen + β L_Recons + γ L_Rep`. The paper establishes the multi-objective form but does not provide a sensitivity analysis over these weights in the reported table. Gradient competition is plausible: generation wants hidden states predictive of next tokens, reconstruction wants maximal context recoverability, and retrieval wants geometric separation. The conclusion’s final-register conflict is one concrete manifestation.

## 4. Flexible inference modes

### 4.1 Regular generation

The model consumes a prompt and generates reasoning/answer tokens as an ordinary causal LM. Registers need not be appended unless another capability is requested. GSM8K and BBH test whether unified tuning preserves generative competence.

### 4.2 Self-reason–latent query embedding

For a complex query, the model first generates text reasoning, then appends registers and pools their final hidden states into an embedding. The same execution can return the reasoning response and vector. This may enrich retrieval because the register bank can attend to explicit decomposition. It also makes query embedding stochastic and expensive: Figure 5 varies temperature and generation lengths from 128 to 8,192, and the NQ setup uses 4,096 maximum new tokens.

### 4.3 Document embedding

Documents use a representation instruction and usually a “None” response rather than a long reasoning trace. Registers compress the document; their pooled last hidden states become the retrieval vector. The NQ setup uses 16 maximum new tokens for document embedding, emphasizing the asymmetric query/document cost.

### 4.4 Latent memory-augmented generation

A document is transformed into 128 registers’ per-layer KV cache. Later questions are generated while attending to that compressed cache instead of raw document tokens. This is more expressive than feeding a single pooled vector because every layer receives multiple key/value slots. It remains lossy and fixed-capacity.

## 5. KV cache and hybrid paged attention

### 5.1 Storage calculation (Equation 6)

Conventional document KV bytes equal `2 × L × H_kv × d_h × N × bytes`, for keys and values across layers, KV heads, head dimension, tokens, and element precision. Under Qwen3-1.7B with 28 layers, eight KV heads, head dimension 128, and bfloat16, the paper gives `114,688 N / 1024²` MiB. Replacing `N` with fixed register count `N_r=128` makes document-cache storage independent of original token length.

The normalization and sign are straightforward. The fixed-size claim excludes model weights, metadata, block fragmentation, prefix/dynamic cache, and the compute needed to create the cache. If 128 registers fail to retain essential information, storage is bounded by discarding it.

### 5.2 HPA design

Paged attention preallocates device-memory blocks and maps sequences to noncontiguous pages. HPA stores ordinary prefix and generated-token KV alongside compressed register KV, using Triton operations and a serving layer built on `nano-vllm`. This allows a request to move from generation to register extraction to latent-memory generation without abandoning paged block management.

### 5.3 Comparison boundary

The naive baseline is adapted Hugging Face generation at batch size one on one A100. HPA configures tensor parallel 1, GPU utilization 0.9, `max_num_seqs=16`, and up to `16,000×16` batched tokens. That is an operationally useful before/after, but batching and scheduling differ. The reported “10×” should be interpreted as system-level improvement over the specified naive implementation, not isolated kernel acceleration or superiority over production vLLM.

## 6. Training and experimental design

### 6.1 Training corpus and partial schedule

Sources include GeneralThoughtArchive, DeepSeek-R1 traces, OpenMathReasoning, ReasonEmbed, DR-Tulu-SFT, and ADP Dataset V1. The authors sample 635,752 examples for 1.7B and 516,336 for 4B but run 550 of 2,483 and 440 of 2,016 planned steps—roughly 22% in each case. The main text’s “around 20%” describes executed training, not a 20% corpus listing.

Both models use 32 GH200 accelerators for 40 hours, batch 8 per device, learning rate `5e-5`, no warmup, AdamW, bfloat16, one positive and one negative, 128 registers, and InfoNCE temperature 0.02. Maximum length is 14,336 for 1.7B and 10,240 for 4B. GradCache handles representation loss; gradient accumulation handles generation/reconstruction. Training uses PyTorch 2.9.0+cu128, Transformers 4.57.1, Datasets 4.3.0, and Accelerate 1.11.0.

### 6.2 Evaluation suites

BRIGHT provides twelve reasoning-intensive retrieval domains and nDCG@10. GSM8K and 23-task BBH cover generation. PwC and 2,952 Wikipedia markdown documents cover reconstruction. Wikipedia articles dated January–March 2026 are intended to reduce contamination, but other training/evaluation overlap is not audited. NQ uses 500 randomly selected test examples against 2,681,468 BEIR NQ documents and a string “match” metric.

### 6.3 Hardware and statistics

BRIGHT uses eight A100 80GB; reconstruction and generation use two RTX PRO 6000 GPUs; RAG uses eight A100 80GB; latency uses one A100 80GB. Seeds, variance, confidence intervals, and exact sample order are not reported. This limits claims about small score differences.

## 7. Results: retrieval and generation

### 7.1 BRIGHT (Table 1)

GRC-1.7B averages 25.7 and GRC-4B 27.2 nDCG@10 at temperature 0.6 with 4K reasoning tokens. ReasonIR-8B reports 24.4, Qwen embedding 22.5, and GritLM-7B 21.0 in the table. GRC-4B leads most baselines on average without external tools, with particularly high TheoremQA-question retrieval at 44.2 and LeetCode at 38.3. It is weak on Pony at 4.1 and AoPS at 11.3.

The smaller 1.7B leads 4B on biology, earth science, Pony, and sustainable living, so scaling is not monotonic per domain. The comparison mixes architectures, training data, query limits, and reasoning compute. It establishes a competitive system result, not parameter-efficiency causality.

### 7.2 Reasoning length and temperature (Figure 5)

Scores generally improve as maximum generated reasoning grows, especially in biology, psychology, and earth science, then plateau. Temperature varies from 0.2 to 0.8. The paper also reports better pooling without EOS. This supports reasoning-conditioned embeddings but introduces two missing controls: a matched-token query-rewrite retriever and repeated stochastic samples with uncertainty.

### 7.3 Generative tasks (Table 4)

GRC-1.7B scores 61.79 on GSM8K and 43.03 on BBH; GRC-4B scores 67.40 and 64.27. The 4B result exceeds tabled GritLM-7B’s 57.5/54.8 and Mixtral 8×7B Instruct’s 65.0/55.9, though benchmark prompting and contamination controls are not detailed enough for a definitive cross-paper ranking. The result does show unified tuning did not destroy reasoning generation.

## 8. Results and ablations: compression, RAG, and serving

### 8.1 PwC reconstruction (Table 2)

GRC-1.7B reports 18.34 sacreBLEU, 0.41 ROUGE-1, 0.25 ROUGE-2, 0.31 ROUGE-L, 0.38 ROUGE-Lsum, and 47.37 chrF. GRC-4B reaches 38.06, 0.63, 0.48, 0.53, 0.60, and 66.63. ICAE-v2-7B reaches 17.98 sacreBLEU and 44.67 chrF but higher ROUGE than GRC-1.7B; GRC-4B leads all listed metrics. PwC training data is not used, making this an out-of-domain claim.

### 8.2 Wikipedia length analysis (Figure 6 and Table 8)

The 2,952-document set spans four bins below 1,024 GPT-2 tokens, with 962/755/719/333 documents in increasing bins. Reconstruction declines with length, especially sacreBLEU and chrF; 4B consistently exceeds 1.7B; high temperature hurts long inputs. Because the range stops below 1,024, this is not yet evidence for very-long-document fidelity despite the long training sequence limits.

### 8.3 NQ RAG (Table 3)

GritLM-7B scores 21.4 without documents and 31.6 with regular documents. GRC-1.7B scores 26.2 no-document, 29.0 compressed, and 33.6 plain. GRC-4B scores 35.4, 40.4, and 45.8. Compressed KV improves answer matching over no document, demonstrating retained utility, but leaves a consistent gap to original text. The `O(1)` versus `O(N)` cache distinction is therefore a rate–distortion trade, not a costless substitution.

### 8.4 HPA latency (Table 15 and Figure 7)

For regular generation/query embedding at 1,024 tokens, 1.7B falls from 32.36 seconds naive to 2.80 HPA; 4B from 41.18 to 3.59. Document embedding stays around 3.8/5.0 seconds naive and 0.33/0.43 HPA because maximum generation length does not affect its short output. Regular RAG at 1,024 falls 24.92→2.21 for 1.7B and 31.36→2.62 for 4B. Latent RAG falls 31.25→2.49 and 39.45→3.22.

These are large system-level gains. The table’s HPA path can batch up to sixteen sequences while naive is batch one; average latency and throughput interactions are not fully separated. A fair serving comparison should include optimized vLLM and equal concurrency/service-level constraints.

## 10. External primary-source vetting

### 10.1 Version and publication status

The live arXiv record is v2, submitted 9 May and revised 12 May 2026. It confirms the three authors, cs.CL classification, and an arXiv-issued DOI. Revision comments specify typo fixes to Equation 4 and GPU names plus HPA implementation details.[^1] No peer-reviewed venue is named.

### 10.2 Code availability

The full-paper HTML contains `https://github.com/gpgg/grclm`, but that address returned 404 during review.[^3] Consequently, loss-temperature semantics, mask code, checkpoints, data sampling, HPA kernels, and evaluator defaults could not be independently audited. Referenced upstream projects such as GradCache, GritLM, Transformers, and nano-vllm do not substitute for the missing GRC implementation.

### 10.3 BRIGHT provenance

The canonical BRIGHT record describes 1,384 real-world queries across twelve reasoning-intensive domains and identifies nDCG@10 as the main retrieval measure.[^4] It reports that adding explicit query reasoning can improve retrieval, which supports GRC’s conceptual direction. The live BRIGHT leaderboard now contains much stronger later systems, so GRC’s table should be read as a paper-time baseline comparison, not a current state-of-the-art claim.

### 10.4 GritLM context

GritLM’s primary record and repository verify that it unifies generation and embedding and provides code with a release pinned for paper reproduction.[^5] GRC’s criticism about distinct modes/masks is a specific architectural comparison; GritLM’s repository also shows that one implementation can load both capabilities, so “two replicas are required” depends on selected class/mode and serving integration rather than a theoretical impossibility.

### 10.5 Associated DEP records

Exact ID, normalized title, and code-link searches across current Black Lake DEP-A/DEP-E indexes and relevant Black-Lake-Data entries found no same-paper or directly corresponding implementation DEP.

## 11. Research notes and critical considerations

### 11.1 Temperature ambiguity

Equation 4 appears to multiply cosine similarity by `τ`, while the table calls 0.02 the InfoNCE temperature. Conventional implementations often divide by 0.02 or multiply by inverse temperature 50. The code is unavailable. This is a reproduction blocker because the two choices produce radically different gradient scales.

### 11.2 Partial training and cost

The models run about one-fifth of planned steps but consume 1,280 GH200 GPU-hours each. The paper does not show learning curves or why those checkpoints were selected. A claim of reduced training cost needs a matched separate-model baseline and equivalent aggregate quality.

### 11.3 Reasoning cost belongs in retrieval accounting

A 4K-token reasoning trace per query may dominate embedding and retrieval latency. BRIGHT quality should be plotted against generated tokens, joules, and wall-clock time, with deterministic non-generative encoders and query rewrite plus standard embedding as controls. HPA reduces serving overhead but not the semantic compute implied by thousands of generated tokens.

### 11.4 Fixed latent capacity

One hundred twenty-eight register slots create a clear storage budget. As source length and information density rise, the ratio tightens. Figure 6 already shows reconstruction decline below 1,024 tokens. Adaptive registers or query-conditioned compression may improve utility but would weaken the simple fixed-size guarantee.

### 11.5 No formal limitations section

The paper’s only explicit methodological limitation appears in the conclusion: the final register hidden state may serve both embedding pooling and next-token prediction, creating conflict. Additional limitations—code absence, stochastic reasoning, contamination, uncertainty, privacy, and memory update semantics—must be inferred rather than collected by the authors.

### 11.6 “Updatable memory” is not demonstrated

The abstract calls the compressed cache updatable, but experiments create and reuse document caches; they do not demonstrate in-place edits, merge policies, temporal supersession, or deletion. A fixed KV block is reusable state, not yet a governed memory database.

## 12. Limitations, implications, and applicability

### 12.1 Core limitations

The method has fixed latent capacity, partial training, missing code, no seed/statistical reporting, possible contrastive-temperature ambiguity, limited compression lengths, and unmatched baseline compute. It evaluates only two Qwen3 base sizes. Compressed RAG remains below raw-text RAG.

### 12.2 Scientific implications

The paper suggests that internal representations can be designed as an interface shared by language modeling, retrieval geometry, and reusable KV state. This is more substantive than merely placing multiple heads on a backbone because reconstruction forces a causal information bottleneck. The result invites analyses of what each register/layer stores and how objectives compete.

### 12.3 Systems implications

If reproduced, one resident model and cache server could simplify RAG deployments that need answers, embeddings, and latent document reuse. HPA highlights that model architecture and serving architecture must be co-designed. Yet conventional embeddings may still win on high-throughput query cost, and raw text may be preferable when fidelity matters more than cache size.

### 12.4 Governance implications

Latent caches can preserve sensitive information without human-readable text, complicating inspection, deletion, and incident response. Operators need source-to-cache lineage, tenant isolation, cryptographic erasure, retention policies, and tests for reconstructability of protected data. A latent representation is not anonymization.

## 13. Falsifiable hypotheses

### Hypothesis 1: utility-aware register allocation dominates a fixed 128 slots

**Test:** allocate 32–512 registers from document length and estimated information density while charging cache bytes.  
**Prediction:** adaptive allocation improves NQ match and long reconstruction at the same average storage.  
**Falsifier:** fixed 128 matches the Pareto frontier across domains.

### Hypothesis 2: latent utility can improve without lexical reconstruction

**Test:** add answer-evidence and contrastive objectives while reducing exact reconstruction weight; compare lexical metrics and RAG.  
**Prediction:** RAG improves even if sacreBLEU declines, showing task-sufficient compression.  
**Falsifier:** RAG tracks exact reconstruction monotonically.

### Hypothesis 3: reasoning-token gains are mostly query rewriting

**Test:** compare GRC self-reason embedding with its generated reasoning text fed to a frozen conventional embedder at matched tokens.  
**Prediction:** much of the BRIGHT gain is recovered by the external embedder, while latent pooling adds a smaller residual.  
**Falsifier:** GRC registers substantially outperform embedding the same trace.

### Hypothesis 4: separating the final register from next-token prediction stabilizes embeddings

**Test:** reserve one pooling-only register or stop its LM-head gradient; measure retrieval variance and generation.  
**Prediction:** retrieval stability improves without material generation loss.  
**Falsifier:** no effect or degraded unified performance.

## 14. Deployment and governance considerations

### 14.1 Deployment controls

- Pin model, data, prompts, mask, register count, loss weights, temperature convention, and serving revisions.
- Benchmark query quality per generated token and total wall-clock time.
- Compare HPA with an optimized vLLM baseline at equal concurrency and latency objectives.
- Monitor compressed-vs-plain answer gaps, reconstruction by length, and cache allocation/failure rates.
- Preserve raw-source references so latent caches can be invalidated and rebuilt.

### 14.2 Safety controls

- Apply access policy before compression; do not treat latent KV as de-identified.
- Propagate deletion to every register-cache replica and block page.
- Test prompt injection persistence through compressed memory.
- Audit cross-tenant page isolation and stale block reuse.
- Require retrieved-source grounding for consequential answers.

## 15. Replication and falsification agenda

### 15.1 Source release requirements

Recover or publish the GRC repository with a pinned commit, license, checkpoints, masks, position-ID logic, loss weights, actual temperature implementation, training manifests, random seeds, HPA Triton code, and evaluator scripts. Checkpoint hashes and exact v2 correspondence are essential.

### 15.2 Training reproduction

Recreate both sampled corpora, report deduplication/contamination checks, run the published partial steps and full planned schedule, and log each loss plus gradient conflicts. Compare unified training with three separate models at matched total accelerator hours.

### 15.3 Evaluation reproduction

Run BRIGHT with deterministic and stochastic seeds, export per-query results, and compare matched reasoning-token baselines. Reproduce GSM8K/BBH prompts, PwC metrics, Wikipedia sampling dates, and NQ 500-example IDs. Add confidence intervals and full-set NQ.

### 15.4 Serving reproduction

Compare naive Hugging Face, HPA, nano-vllm, and production vLLM at batch 1 and controlled concurrency. Report p50/p95/p99 latency, throughput, memory, queueing, cache-hit rate, and energy for each mode. Separate prefill, generation, register extraction, and latent-cache attention.

### 15.5 Falsification criteria

The unification claim weakens if a single-objective or separate-model baseline provides better aggregate quality at matched compute and memory. The compression claim fails operationally if fixed caches cannot preserve answer evidence across realistic long documents. The HPA claim narrows if gains disappear against an optimized equal-concurrency baseline.

## 16. Durable restatement

GRC uses continuous meta-latent registers as a shared boundary between three jobs. A customized causal mask lets registers read source context while forcing a reconstruction segment to recover it through the registers; the same register hidden states are pooled into retrieval embeddings, and their per-layer KV can be stored as fixed-length latent memory for later generation. The paper demonstrates promising retrieval, generation, reconstruction, RAG, and serving results at 1.7B and 4B scale. Its central lesson is that unified representation learning must be paired with cache-aware serving. Its central caution is equally important: generated reasoning is costly, fixed latent memory is lossy, and reproducibility remains blocked while the paper-linked implementation is unavailable.
