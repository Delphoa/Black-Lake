# Vertical Chunking as a Memory Schedule for Recurrent Text Embedders

## A detailed review, technical reconstruction, and independent re-conceptualization of “Linear-Time and Constant-Memory Text Embeddings Based on Recurrent Language Models”

**Paper:** Tobias Grantner, Emanuel Sallinger, and Martin Flechl, *Linear-Time and Constant-Memory Text Embeddings Based on Recurrent Language Models*, arXiv:2604.18199v1, 20 April 2026.[^1]  
**Canonical record:** https://arxiv.org/abs/2604.18199  
**Reviewed evidence:** locally archived full PDF and full-paper HTML; canonical arXiv record; author-released models and inference components; official MTEB/MMTEB sources; and the LongEmbed record.[^2]  
**Coverage:** main text, limitations, acknowledgments, Appendices A–G, thirteen tables, ten figures, and Equations 1–12.  
**Reproduction status:** source and code availability were vetted, but training and inference experiments were not rerun.

---

## Executive assessment

### The paper in one sentence

The authors turn pretrained recurrent language models into general-purpose text embedders and introduce a cross-layer execution schedule that processes a fixed vertical token block through every layer before advancing, making activation memory independent of total input length once the input exceeds that block.

### The deeper idea

The most important contribution is not simply “use Mamba for embeddings.” It is a scheduling transformation. A recurrent layer can process arbitrarily long streams with a fixed state, but modern implementations recover hardware parallelism by evaluating token chunks with matrix operations. Standard layer-by-layer execution keeps a full sequence of intermediate activations alive between layers, bringing back memory growth with sequence length. Vertical chunking changes loop order: process token block 1 through all layers, retain one recurrent state per layer, then process token block 2. This caps live activations at the chosen vertical block size.

The paper combines this schedule with an E5-style contrastive recipe. It removes each language model’s prediction head, appends an end-of-sequence token, and uses the final hidden state as the embedding. Fine-tuned Mamba2 models are close to similarly sized transformers on MTEB and LongEmbed, although the direction changes by benchmark: Codestral Mamba2 7B leads the matched Mistral baseline on multilingual MTEB, trails it on English MTEB, and is nearly tied on LongEmbed.

### Bottom-line judgment

The scheduling result is technically credible and practically interesting. The paper derives the Mamba2 recurrence/matrix duality, identifies the activation-lifetime problem across layers, and tests the proposed loop interchange on Mamba2, RWKV7, and xLSTM. Main H100 measurements and cross-device appendices report the expected pattern: memory grows until the vertical chunk size, then remains essentially flat; runtime approaches horizontal execution once the chunk is large enough to saturate the accelerator. Author-released checkpoints and chunkable inference components were present in a public Hugging Face collection at inspection.[^3]

The title requires precise interpretation. “Constant-memory” is with respect to input length after `T > V`, for fixed model depth, state size, batch size, precision, and vertical chunk size.[^6] It is not constant total memory across models or batches. “Linear-time” follows when the intra-layer chunk size `Q` is fixed; the derived work is `O(TQ)`, not an input-independent constant factor. The quality comparison is also not architecture-pure because recurrent and transformer base models differ substantially in pretraining amount and domain. The authors disclose this and correctly present recurrent embeddings as viable, not universally superior.

### Principal strengths

- A clear formal bridge connects recurrence, structured matrix computation, chunk-local work, and cross-layer scheduling.
- The model-quality comparison uses a largely shared contrastive recipe and reports multiple model sizes and benchmark families.
- The efficiency study covers single and batched inputs, three recurrent families, two transformer variants, and three accelerator types.
- Appendices disclose data cleaning, training hyperparameters, learning-rate tuning, task-type results, prompts, and hardware robustness.
- The code/model collection exposes both checkpoints and adjusted Mamba2/sentence-transformer components.
- Limitations openly identify pretraining mismatch, untested recurrent embedding families, missing input-length quality controls, and bias risks.

### Principal qualifications

- English retrieval shows a meaningful quality gap: Codestral Mamba2 7B scores 50.3 versus 56.8 for Mistral 7B v0.1 in the detailed English v2 table.
- Exact runtime and memory values are plotted rather than supplied in a machine-readable numeric table in the paper.
- Three repetitions address system variability only weakly; no statistical model for timing or benchmark uncertainty is reported.
- LongEmbed is evaluated to 32,768 tokens, but quality is not stratified by actual document length.
- Codestral is a code-oriented base model with undisclosed pretraining details, limiting causal attribution to architecture.
- Constant activation memory does not include an unbounded output store, corpus index, or model weights; it concerns one embedding forward pass.

## 1. Problem, stakes, and prior approaches

### 1.1 Why long-document embedding is costly

Embedding systems map variable-length text to fixed-size vectors for retrieval, clustering, classification, similarity, and reranking. Transformer encoders can provide strong representations, but full self-attention has quadratic computation in sequence length. Memory-optimized kernels such as FlashAttention avoid materializing the full attention matrix, yet intermediate activation storage still grows with the token sequence. Sliding-window attention makes attention cost linear but limits long-range interaction.

For document indexing, these costs matter differently than autoregressive generation. The system must encode every token before pooling a single vector. Long legal, scientific, or operational documents can exceed memory even though the output is small. A recurrent model is attractive because it maintains a fixed state while scanning tokens. The systems challenge is to preserve accelerator parallelism without retaining full-sequence activations between dozens of layers.

### 1.2 The gap addressed

Modern recurrent architectures such as Mamba2, RWKV, and xLSTM offer recurrence plus parallelizable formulations. Prior work emphasizes language modeling and decoding. The paper asks two connected questions: can these pretrained models become competitive general-purpose embedders under a standard contrastive recipe, and can their recurrence be scheduled across layers to keep embedding inference memory bounded in sequence length?

### 1.3 What is inherited and what is novel

EOS pooling, InfoNCE, instruction-conditioned queries, QLoRA, MTEB, and LongEmbed are inherited. Mamba2’s structured state-space duality is inherited from Dao and Gu. The authors’ novelty is applying the E5-style training recipe to Mamba2 bases, deriving the relevant chunk decomposition pedagogically, introducing vertical cross-layer chunk execution, implementing it, and empirically showing that the scheduling principle also holds for RWKV7 and xLSTM.

## 2. Embedding training reconstruction

### 2.1 EOS pooling (Equation 1)

For token sequence `s = (t1,…,tn)`, the model appends its EOS token. The language-model head is removed, and the final-layer hidden vector at EOS becomes `E(s)`. Because a causal model’s terminal position can depend on every preceding position, it acts as a learned summary. The output dimension is fixed even as input length varies.

This pooling is simple but creates a directional bottleneck: all information must reach the last state. Recurrent models are structurally compatible with that flow, while a bidirectional encoder might use mean or special-token pooling. The paper does not compare pooling alternatives, so quality claims attach to the whole recipe rather than recurrence alone.

### 2.2 InfoNCE objective (Equation 2)

Each batch contains queries, positives, and zero or more hard negatives. The loss minimizes the negative log probability assigned to the positive under exponentiated cosine similarity divided by temperature `τ`. All positives and hard negatives associated with other batch queries also become in-batch negatives. With `τ = 0.02`, small cosine differences are sharpened considerably.

The numerator and denominator are normalized consistently; the objective increases positive similarity relative to every negative. It does not guarantee calibrated cosine values across tasks. Instruction tuning is implemented by prefixing the query—not documents—with `Instruction: {prompt}` and `Query:`. A corpus can therefore be embedded once and searched using task-conditioned queries.

### 2.3 Data and optimizer details

The combined training collection contains 1,739,369 triples before filtering. The authors find 15,726 triples, 0.90%, with at least one empty query, positive, or negative string and remove them. Public datasets contribute 1,248,378 nonempty triples; several synthetic categories contribute the remainder. All models train for one epoch with batch size 2,048, 100 warm-up steps and linear decay, weight decay 0.1, maximum length 512, and QLoRA using four-bit weights, bfloat16 activations, rank 16, and alpha 16 on all linear layers.

Most models use learning rate `4×10^-4`. Mamba2 1.3B requires a separate sweep on one-sixteenth of the data; `2.4×10^-3` gives the best reported English v2 task mean and is used for the final run. This model-specific tuning is reasonable, but it means “identical conditions” should be read as a shared pipeline with a tuned learning rate, not literally identical hyperparameters.

## 3. Recurrent and matrix views of Mamba2

### 3.1 Linear recurrence (Equations 3a–3b)

At step `t`, Mamba2 updates a state `h_t = A_t h_(t-1) + B_t x_t` and emits `y_t = C_t^T h_t`. With state size fixed, scanning a length-`T` input costs `O(T)` work and `O(1)` state memory with respect to `T`. The catch is dependency: `h_t` must follow `h_(t-1)`, restricting token-parallel execution.

### 3.2 Structured unrolling (Equations 4–6)

Mamba2 constrains `A_t` to scalar `a_t` times identity. Products of transitions can therefore be represented compactly. Expanding the recurrence expresses each output as a sum of all prior inputs after transition, input, and output transforms. The whole sequence becomes `y = Mx`, where `M` is lower triangular because future tokens cannot affect past outputs. A causal kernel `L` contains scalar transition products, and `M = L ∘ (CB^T)` uses elementwise multiplication.

This form admits efficient matrix multiplication and is central to parallel training. Materializing the full `T×T` operator, however, creates quadratic work and memory. The duality is not a free reduction: recurrence is memory-efficient but sequential; the matrix form is parallel but can be quadratic.

### 3.3 Complexity boundary

The paper’s inference scheme chooses an intermediate point by fixing an intra-layer chunk size `Q`. Work becomes `O(TQ)` and activation memory for horizontal chunked execution follows the same sequence-linear scaling. When `Q` is treated as a deployment constant—256 for Codestral in the experiments—this is linear in `T`. If `Q` grew with `T`, the asymptotic statement would change.

## 4. Chunked inference and vertical scheduling

### 4.1 Intra-layer chunks (Equations 7–12)

The sequence is partitioned into `K = T/Q` chunks. Equations 7a–7f define chunk-indexed views of the causal kernel, transitions, `B`, `C`, and input. A boundary state `b^(c)` equals the recurrent state after chunk `c`.

The computation has three stages. First, every chunk computes its local output and local boundary contribution under a zero incoming state using matrix operations (Equations 8–9). Second, the boundary recurrence propagates the prior final state into the current chunk (Equation 10). Third, an inter-chunk correction based on the incoming state is added to each local output (Equations 11–12). Only the `K` boundary updates remain sequential; most chunk-local work can be parallelized.

The signs are additive because the state equation is linear in the previous state and current input. The lower-triangular/causal structure prevents future-chunk leakage. The paper omits discretization and multidimensional detail in the exposition, explicitly delegating the complete Mamba2 proof to prior work; the presented derivation is a scheduling foundation, not a new equivalence theorem.

### 4.2 Horizontal execution

Ordinary layer-major execution processes the entire token sequence through layer 1, then the entire sequence through layer 2, and so on. Within each layer, `Q`-sized matrix chunks provide parallelism. But the next layer needs the full output sequence, so live activations scale as `O(TQ)` under the authors’ accounting. Figure 6 expands this layer-by-layer accumulation.

### 4.3 Vertical execution

Vertical chunking introduces a second block size `V`, a multiple of `Q`. Token block 1 of length `V` passes through all `L` layers before token block 2 begins. The system stores activations only for the active block and one recurrent state per layer. Figure 7 visualizes this changed traversal order.

For `T > V`, reported memory is `O(L + QV)`: `L` state objects plus current-block activations. With fixed `L`, `Q`, and `V`, memory no longer grows with `T`. For `T ≤ V`, the implementation can use ordinary horizontal execution and avoid maintaining cross-block state. Vertical chunking therefore changes scheduling, not the mathematical embedding, assuming state passing is implemented exactly.

### 4.4 Choosing `Q` and `V`

`Q` balances more token parallelism against quadratic within-chunk work. Codestral performs best at `Q = 256` in the reported sweep. `V` balances cross-layer block size against memory. For batch size greater than one, the authors find throughput nearly saturated around `V = Q` and fully around `2Q`. For batch size one, roughly `V = 4096` is needed to saturate the tested devices. These are empirical guidelines, not architecture-independent constants.

## 5. Architecture and pipeline

### 5.1 Training pipeline

The system cleans and samples contrastive triples, formats instructions only on queries, tokenizes to at most 512 positions, appends EOS, runs the causal base, extracts its final hidden vector, and optimizes InfoNCE through QLoRA adapters. A dataset-proportional sampler draws each batch from a single source dataset because the original reproduction did not expose a precise sampling strategy.

### 5.2 Inference pipeline

For short sequences, the model may run horizontal chunked inference. For longer sequences, it chooses `Q` and `V`, initializes one recurrent state per layer, passes each vertical block through all layers while updating those states, obtains the EOS hidden state at the end, and returns it as the document vector. Batched documents follow the same schedule but saturate hardware at smaller `V`.

### 5.3 Reproducible artifact boundary

The live author collection contained 7B and 1.3B model entries, a chunkable Mamba2 code repository, and a chunkable sentence-transformer component.[^3] A complete reproduction still requires exact collection revisions, model hashes, dependencies, CUDA/kernel versions, benchmark revision, task instructions, random seeds, and the figure-data extraction path.

## 6. Experimental design

### 6.1 Quality benchmarks

The evaluation covers MTEB English v1/v2, MTEB Multilingual v2, and LongEmbed. English v2 comprises 41 datasets across classification, clustering, pair classification, reranking, retrieval, STS, and summarization. Multilingual v2 spans more than 250 languages. LongEmbed contains two synthetic and four real-world long-retrieval tasks and is run to 32,768 tokens here.[^4]

MTEB evaluations use bfloat16 and cap sequences at 512 for comparability. LongEmbed uses the Mistral-family 32,768-token limit. Aggregate results report both means over tasks and, where applicable, means over task types. Those denominators are not interchangeable: a task mean weights datasets, while a task-type mean gives categories more equal influence.

### 6.2 Model controls

At roughly 1B scale, Mamba2 1.3B is compared with Qwen2 1.5B. At 7B, Codestral Mamba2 is compared with Mistral 7B v0.1; v0.3 is also trained for inference comparison. The same broad fine-tuning recipe improves or closely reproduces earlier transformer results, providing a useful implementation control.

The comparison cannot isolate architecture from pretraining. Mamba2 1.3B saw about 700B pretraining tokens versus Qwen2 1.5B’s reported 7T. Codestral’s exact corpus is unavailable and is code-oriented. This confounding can plausibly explain benchmark-specific differences.

### 6.3 Efficiency protocol

The main runtime and memory study uses one 80GB H100, bfloat16, varying input lengths and batch sizes, and three repetitions with means. Mamba2 experiments vary `Q` and `V`; RWKV7 and xLSTM use their chosen native/saturating chunk configurations; Mistral uses FlashAttention-2. Appendix E repeats the main patterns on L40S, A100, and H100, reporting mean/min/max plots.

## 7. Results: quality

### 7.1 Aggregate comparison (Table 1)

At the smaller scale, Qwen2 1.5B versus Mamba2 1.3B scores are: multilingual task mean 56.2 versus 55.2, multilingual task-type mean 48.7 versus 47.9, English v2 task mean 65.7 versus 64.3, task-type mean 62.3 versus 60.9, and LongEmbed 41.0 versus 40.8. The recurrent model is close but consistently lower.

At 7B, Mistral v0.1 versus Codestral Mamba2 scores are: multilingual 58.6 versus 59.4 by task and 50.5 versus 51.9 by task type; English 67.3 versus 65.2 and 63.3 versus 61.8; LongEmbed 44.7 versus 44.5. “Competitive” is supported. “Better” would require a benchmark-qualified statement.

### 7.2 Training-recipe control (Table 2)

On English MTEB v1, the paper’s Qwen2 reproduction reaches 63.6 task mean and 61.8 task-type mean, compared with Springer et al.’s 63.3/61.3. Its Mistral v0.1 reaches 65.7/63.5 versus the earlier 65.5/63.2. These small improvements support pipeline fidelity but do not establish exact replication because data cleaning and sampling details differ.

### 7.3 Detailed task behavior (Tables 4–7)

On multilingual v2, Codestral leads Mistral v0.1 in bitext mining by 2.0, reranking by 4.8, and retrieval by 1.9 points. On English v2 the pattern reverses, especially retrieval: 50.3 versus 56.8, a 6.5-point deficit. Other English task-type gaps are closer. On LongEmbed, each 7B model leads on three of six tasks; aggregate means differ by 0.2. English v1 also favors Mistral overall.

The granular results are more informative than a single mean. They suggest recurrent embedding viability but not task-invariant parity. The multilingual advantage may reflect Codestral pretraining, tokenizer behavior, or the architecture; the experiment cannot separate these explanations.

## 8. Results: inference and ablations

### 8.1 Intra-layer chunk size (Figure 3)

Codestral runtime is swept across `Q` for batch sizes 1 and 128. The paper reports `Q = 256` as best across tested lengths and batches. Smaller chunks underutilize parallel hardware; larger chunks pay more quadratic within-chunk cost. This is the expected U-shaped tradeoff and justifies fixing 256 for subsequent Mamba2 experiments.

### 8.2 Vertical chunk size (Figure 4)

With `Q = 256`, batch-one runtime approaches horizontal execution as `V` increases and converges at 4,096 in the tested H100 setup. Batch 32 stays close to horizontal runtime even at small `V`, with only a minor penalty at 256. Memory grows to the chosen `V` then plateaus for longer inputs, providing the direct empirical support for the constant-in-`T` claim.

### 8.3 Architecture comparison (Figure 5)

For batch one beyond 4,096 tokens, Codestral Mamba2 is reported faster and lower-memory than both Mistral variants. Mistral v0.3 uses full attention and develops quadratic runtime growth; v0.1 uses a 4,096-token sliding window, avoiding that growth but limiting global attention. At batch 32, runtimes are broadly comparable, while transformer runs beyond 4,096 fail the 80GB constraint. Mamba2, RWKV7, and xLSTM share the memory plateau pattern under vertical scheduling.

### 8.4 Cross-hardware robustness (Figures 8–10)

L40S, A100, and H100 plots retain the same qualitative findings: native `Q` remains suitable, batch-one saturation requires up to roughly 4,096 tokens, batched saturation occurs near `Q` or `2Q`, and recurrent long-sequence memory remains favorable. Hardware portability is supported qualitatively; exact cost ratios remain device- and kernel-specific.

## 10. External primary-source vetting

### 10.1 Identity and publication status

The arXiv record confirms the title, three authors, cs.CL classification, v1 date, PDF/HTML/source availability, and arXiv-issued DOI.[^1] No accepted venue is named there; this review treats the work as a preprint.

### 10.2 Code and checkpoint status

The paper’s shortened footnote resolves to an author-controlled Hugging Face collection. At inspection it listed a 7B embedding model, a 1.3B embedding model, adjusted Mamba2 code/kernels for chunked state passing, and a chunkable sentence-transformer component.[^3] This is materially stronger evidence than a paper-only code promise. It does not by itself verify the published scores.

The official MTEB paper defines a multi-task embedding evaluation and explicitly observes that no one method dominates all tasks. Current official documentation identifies MTEB and MMTEB benchmark families and their task metadata.[^4] The canonical LongEmbed record describes two synthetic and four real-world long-retrieval tasks and context extension to 32K.[^5] These sources corroborate the paper’s benchmark descriptions while reinforcing that aggregate means span heterogeneous tasks.

### 10.4 Associated DEP records

Exact arXiv ID, title, and released-artifact searches across the live Black Lake DEP-A/DEP-E indexes and relevant Black-Lake-Data deposition tree found no same-paper or direct implementation DEP. No association is asserted from generic recurrence or embedding similarity.

## 11. Research notes and critical considerations

### 11.1 The real novelty is loop interchange

Vertical chunking changes the order of token-block and layer loops while carrying recurrent states across blocks. This makes activation lifetime explicit and suggests a reusable optimization pattern for any stacked recurrent architecture with exact state continuation. The requirement of a parallel chunk form distinguishes it from a purely sequential RNN implementation.

### 11.2 Constant-memory language must stay conditional

The relevant independent variable is total sequence length. Memory can still grow with batch, depth, hidden/state dimension, `Q`, `V`, precision, and implementation workspace. Plots show “practically constant” device memory beyond the vertical block, which may include allocators and kernels not captured by the asymptotic model. Deployment documentation should state the fixed variables beside every plateau claim.

### 11.3 Quality confounding

The smaller recurrent base has one-tenth the reported pretraining tokens of Qwen2 1.5B. The larger recurrent base is code-specialized with undisclosed corpus details. Architecture, data, tokenizer, and pretraining objective all vary. The study is sufficient to show viable checkpoints, but not to estimate a causal recurrent-versus-transformer quality effect.

### 11.4 Evaluation denominators

Task means and task-type means answer different questions. Multilingual v2 includes uneven dataset and language coverage, while English retrieval accounts for much of the 7B gap. A deployment should select tasks and length distributions matching its corpus instead of using the global mean as a universal score.

### 11.5 Missing controlled length-quality curve

The efficiency curves vary token length; the quality benchmarks do not report quality binned by length. A recurrent model may preserve access to early information differently from full attention or sliding windows. LongEmbed aggregate parity is encouraging but does not locate where failures occur.

### 11.6 Training-data and funding provenance

The appendices disclose empty-triple cleaning, synthetic/public composition, instruction counts, and core hyperparameters. The acknowledgment states funding for Emanuel Sallinger from WWTF and FWF under listed grants. No code/data reproduction claim should omit those provenance and support statements.

## 12. Limitations, implications, and applicability

### 12.1 Limitations

Only Mamba2 receives embedding-quality evaluation; RWKV7 and xLSTM appear only in efficiency tests. Pretraining is unmatched. Quality is not controlled by input length or specialized domain. Runtime evidence is accelerator-specific and uses three repetitions. Exact figure coordinates are not supplied as tables. The authors also note that embedding similarities can reproduce biases from pretraining data.

### 12.2 Scientific implications

Embedding architecture need not inherit the transformer monopoly of current toolchains. A recurrent foundation plus conventional contrastive training can be broadly useful. More importantly, evaluation should distinguish representation quality from execution schedule: the same trained recurrent model can run horizontally or vertically, permitting exact system-level comparisons.

### 12.3 Deployment fit

The method fits long-document indexing, log/event summarization into vectors, and batched retrieval corpora where accelerator memory is the binding constraint. It is less compelling for short strings, where transformer kernels are mature and vertical state management adds no benefit, or where a domain-specific small encoder already meets quality and latency requirements.

### 12.4 Governance

Efficient embeddings make large-scale indexing easier, which increases privacy and bias exposure. Operators need data minimization, access controls, deletion propagation, model/version records, and domain-specific retrieval audits. Cosine similarity is not a factual or fairness judgment; high similarity should not automatically drive consequential decisions.

## 13. Independent hypotheses

### Hypothesis 1: vertical scheduling is numerically equivalent within tolerance

**Test:** run identical weights and inputs through horizontal and vertical schedules across `Q`, `V`, precision, and architecture, comparing EOS vectors and retrieval rankings.  
**Prediction:** bfloat16 differences remain small enough that top-k ranking is stable for almost all queries.  
**Falsifier:** systematic rank changes grow with the number of vertical blocks.

### Hypothesis 2: length-conditioned contrastive training closes the English retrieval gap

**Test:** mix 512-token training with long positive/negative documents and compare to the paper’s checkpoint on English retrieval bins.  
**Prediction:** Codestral’s 6.5-point English retrieval deficit shrinks without erasing multilingual gains.  
**Falsifier:** long training fails to improve retrieval or degrades other task types beyond the gain.

### Hypothesis 3: an adaptive `V` controller dominates fixed settings

**Test:** choose `V` from current free memory, batch size, and a calibrated throughput model; compare throughput and OOM rate with fixed `Q`, `2Q`, and 4,096.  
**Prediction:** adaptive scheduling increases successful tokens/second under mixed workloads while preserving the memory cap.  
**Falsifier:** fixed heuristics match adaptive throughput and reliability across devices.

### Hypothesis 4: pretraining-matched recurrence narrows architecture uncertainty

**Test:** pretrain recurrent and transformer models on the same tokens, tokenizer, parameter budget, and objective, then apply identical embedding fine-tuning.  
**Prediction:** much of the English gap disappears, while recurrent memory scaling remains.  
**Falsifier:** a stable quality gap persists across seeds and tasks despite matched pretraining.

## 14. Deployment and governance considerations

### 14.1 Operational controls

- Pin model, kernel, CUDA, benchmark, prompt, `Q`, `V`, precision, and tokenizer revisions.
- Validate horizontal/vertical embedding equivalence before rollout.
- Monitor peak allocated/reserved memory, p50/p95/p99 latency, throughput, and OOM rate by input-length bin.
- Maintain a transformer or prior-checkpoint canary for retrieval-quality drift.
- Record embedding versions so indexes can be rebuilt deterministically.

### 14.2 Safety and data controls

- Minimize source documents before embedding and restrict index access by tenant.
- Make deletion apply to raw text, embeddings, caches, and replicas.
- Audit nearest-neighbor behavior across protected and domain-critical slices.
- Require downstream evidence retrieval; do not treat vector proximity as proof.
- Preserve funding, model-card, license, and benchmark provenance in production documentation.

## 15. Replication and falsification agenda

### 15.1 Artifact lock

Snapshot the two author checkpoints and chunkable components with commit/revision hashes. Record environment packages, kernel build, CUDA driver, exact devices, and licenses. Export the MTEB and LongEmbed task revisions and instruction tables used by the paper.

### 15.2 Training reproduction

Reconstruct the 1,739,369 pre-filter triples, verify all 15,726 empty-containing removals, implement single-dataset batches with size-proportional sampling, and rerun one-epoch QLoRA. Repeat the Mamba2 learning-rate sweep and report seeds, full curves, and checkpoint selection.

### 15.3 Quality reproduction

Evaluate all five trained models in bfloat16. Report per-task results, task and task-type means, confidence intervals, and quality by document-length bin. Add frozen-base pooling controls, mean pooling, and a matched-pretraining comparison when possible.

### 15.4 Efficiency reproduction

Test horizontal and vertical schedules at batch 1/32/128 on L40S, A100, and H100. Export numeric CSVs for every figure, including min/mean/max across more than three runs, warmup policy, synchronization, allocated/reserved memory, and failures. Confirm transformer context semantics, especially Mistral v0.1 sliding-window versus v0.3 full attention.

### 15.5 Falsification threshold

The central systems claim fails if memory continues growing materially with `T` after `T > V` under controlled allocation, or if vertical execution changes embeddings enough to alter retrieval results systematically. The broader quality claim weakens if recurrent checkpoints fall consistently behind matched transformers on representative length-stratified tasks.

## 16. Durable restatement

This paper demonstrates two separate but complementary facts. First, pretrained Mamba2 language models can be adapted into useful EOS-pooled text embedders with a conventional contrastive recipe; they are close to transformer controls but have benchmark-specific strengths and weaknesses. Second, stacked recurrent embedders need not retain sequence-length activations between layers. By processing a fixed token block through the full depth and carrying one state per layer, vertical chunking makes live activation memory independent of total sequence length beyond the block while retaining enough matrix parallelism for practical throughput. The durable contribution is therefore a reusable memory schedule, bounded by explicit assumptions—not a claim that recurrence automatically wins every embedding benchmark.
