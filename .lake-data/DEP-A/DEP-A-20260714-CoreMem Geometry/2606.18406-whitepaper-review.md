# CoreMem’s Evaluation Is Useful; Its Geometry and Bound Need Repair

## A full-paper review and independent re-conceptualization of “CoreMem: Riemannian Retrieval and Fisher-Guided Distillation for Long-Term Memory in Dialogue Agents”

**Paper:** Jiaqi Chen, Yongqin Zeng, Shaoshen Chen, Yijian Zhang, Hai-Tao Zheng, Chunxia Ma, and XiuTeng Zhou, arXiv:2606.18406v1, 16 June 2026.[^1]
**Scope:** complete PDF and full-paper HTML, TeX source and appendix, all algorithms, equations, the theorem and proof sketch, three figures, every main and appendix table, prompts, case studies, limitations, AI-use disclosure, and primary-source vetting.
**Review activity:** formal reconstruction, dimension and sign audit, numerical cross-check, evaluation-design review, and public-artifact check. No code, retrieval run, compressor, API call, or benchmark was independently reproduced.

---

## Executive assessment

### The paper in one sentence

CoreMem combines covariance-whitened memory retrieval with gradient-based token pruning and supplies a broad consumer-hardware evaluation, but the manuscript mislabels a global Mahalanobis-style score as locally adaptive Fisher–Rao geometry, contains dimensionally invalid pseudocode and an unproved compression bound, and sometimes overstates small absolute gains.

### Durable contribution

The strongest contribution is the experimental decomposition. The paper separately evaluates retrieval on two dialogue-memory benchmarks, compression under a shared retriever, embedding-size effects, compression budgets, structural heuristics, end-to-end wall time, prompt tokens, and failure cases. It explicitly shows that its compressor loses to simple truncation below 1,000 tokens and that its retrieval can be pulled toward a wrong semantic hub. That candor makes the paper valuable even where its theory fails.

The practical architecture is also coherent: store and search memory on an edge device, compress only retrieved turns, and send the resulting text—not the full history—to a cloud LLM. Because FDTD returns discrete retained tokens rather than latent vectors, operators can inspect what leaves the device.

### Bottom-line judgment

CoreMem-Fusion’s reported accuracy gains are real within the printed tables but modest overall. On LOCOMO with MiniLM, Judge accuracy rises from 0.503 for NaiveRAG to 0.540, a 3.7-point absolute gain. On gte-Qwen2 it rises from 0.531 to 0.536, only 0.5 points. LongMemEval-S rises from 0.436 to 0.462. These are meaningful if statistically stable, and the LOCOMO result carries a reported McNemar `p<0.001`, but multiple system and hyperparameter comparisons increase the risk of selection without correction.

The terminology overclaims the method. The “Riemannian” component estimates one global covariance from all normalized memories, then scores query–memory pairs using an inverse-covariance bilinear form. It is not local per query or memory, and the paper does not derive it as a Fisher–Rao geodesic between distributions. It is best described as low-rank covariance whitening or Mahalanobis-style retrieval.

The appendix pseudocode is not executable as written. For centered matrix `H∈R^(N×d)`, SVD left singular vectors `U[:, :r]` live in item space `R^(N×r)` and cannot be added to diagonal feature covariance `D∈R^(d×d)`. Principal directions must come from `V[:, :r]`. The proposed `P=D^-1 U M^-1` also omits the central Woodbury inverse `(M^-1+UᵀD^-1U)^-1`, and the final product is not the stated covariance inverse. These are structural dimension/algebra defects.

The FDTD “theorem” is not established. Its implementation computes next-token cross-entropy of the local distiller on the context itself, not a target answer loss from the cloud generator. A Lipschitz assumption does not imply that loss change is bounded by the sum of `||grad⊙embedding||`, especially after discrete deletion shifts all later positions. The appendix changes the theorem’s notation and offers an assertion rather than a proof. Structural boosts—not Fisher theory—appear to do much of the practical work.

## 1. Problem framing and scope

The target is long-term personalized dialogue on consumer devices. A memory bank contains `N` historical units with embeddings; a query retrieves top `k`; a local model reduces retrieved text to budget `B`; a cloud model answers. Objectives are answer accuracy, bounded edge compute and VRAM, and reduced cloud prompt cost.

CoreMem has two separable modules:

1. a covariance-aware retriever that interpolates cosine and inverse-covariance scores; and
2. Fisher-Guided Discrete Token Distillation, which scores tokens using gradients from a frozen local causal model, applies hand-built syntax/content multipliers, and selects text under a budget.

The modules are not trained jointly. “Unified by information geometry” is conceptual branding, not an optimization coupling: retrieval uses empirical covariance, while compression uses a first-order gradient magnitude plus heuristics.

## 2. Technical reconstruction

### 2.1 Covariance-aware retrieval

Memory and query embeddings are L2-normalized. The method computes global coordinate variances, adds ridge `λ=10 mean(σ²)`, and performs SVD on the centered memory matrix. The minimal rank explaining 95% variance supplies a low-rank correction. Woodbury identity is intended to avoid a dense `d×d` inverse.

For a valid feature covariance, one would write `Σ=D+V_r M_r V_rᵀ`, with `V_r∈R^(d×r)`. Its inverse is:

`D^-1 − D^-1 V_r (M_r^-1+V_rᵀD^-1V_r)^-1 V_rᵀD^-1`.

The main equation has this form if `U_r` is interpreted as feature directions. The algorithm, however, explicitly takes `U` from `H=USVᵀ`, producing the wrong dimension. It then computes a different shortcut. The implementation must be inspected to determine which version generated results.

The ridge claim is also unsupported. Adding ten times mean variance to the diagonal does not generally bound the condition number near two: maximum coordinate variance can greatly exceed the mean, and the positive low-rank term can enlarge the top eigenvalue. A bound would require explicit constraints on maximum variance and low-rank spectrum.

### 2.2 Score fusion

The paper min–max normalizes cosine and “Riemannian” scores, then computes:

`score = α cosine + (1−α) rie`.

Therefore `α=0.9` strongly favors cosine, not Riemannian retrieval. The manuscript repeatedly calls that setting “strong Riemannian bias,” a direct sign/interpretation contradiction. `α=0` is pure inverse-covariance retrieval and `α=0.5` is balanced Fusion.

Min–max scaling occurs over the candidate memory bank, so adding an outlier can change all fused scores. Constant-score cases and numerical epsilon are not specified.

### 2.3 FDTD token sensitivity

Retrieved text is fed to a frozen local causal model. The algorithm uses the text itself as next-token labels, backpropagates cross-entropy to input embeddings, and defines:

`S_i = ||∇_(h_i)L ⊙ h_i||_2`.

This is a first-order saliency score related to “gradient times input.”[^5] It is not the trace of a Fisher information matrix unless additional expectation and squared-gradient assumptions are defined. The main text calls it a diagonal Fisher trace, while the appendix acknowledges it is computed directly from one gradient.

The target mismatch is fundamental. The local loss asks which tokens help predict the context’s next tokens; the downstream task asks which tokens help a cloud model answer a query. A name or date can be locally predictable yet answer critical, or surprising yet irrelevant.

### 2.4 Structural protection and selection

Scores are smoothed with width-five average pooling, then multiplied by fixed rules: logical keywords 1.2×; punctuation and boundaries 1.3–1.5×; digits and capitalized words 1.3–1.4×; and a turn-depth decay from 1.0 to 0.6. Gaps of at most three tokens are filled.

The prose says the method first splits turns/sentences, protects the first and last three tokens of sentences longer than eight, prunes within sentences, and selects sentences by mean score. The appendix algorithm instead sorts all tokens globally and does not implement the sentence hierarchy. It also fills gaps after selecting `B`, which can exceed the strict budget. The relationship between prose, pseudocode, and actual code is unresolved.

### 2.5 Claimed error bound

The theorem states generation KL is `O(L² Σ_dropped S_i)` under local Lipschitz continuity. The appendix instead bounds target cross-entropy difference by `C ΣS_i`. No derivation connects gradient-times-input to deletion perturbations, KL, cloud-model loss, or the squared Lipschitz constant. Greedy selection is optimal only for additive item weights without gap-filling, sentence constraints, or dependencies.

The correct status is heuristic motivation. A valid analysis would define a continuous deletion path, bound second-order remainder, account for positional shifts, and use gradients of the actual downstream model or a calibrated proxy.

## 3. Architecture and pipeline

The edge encodes memories with either gte-Qwen2-1.5B (1,536 dimensions) or MiniLM-L6 (384). Retrieval returns top 50. FDTD uses Qwen3-0.6B, Llama-3.2-1B, or SmolLM2-360M. GPT-4o-mini generates and separately judges answers at temperature zero.

Figure 1 presents the edge/cloud pipeline and claims about VRAM, time, tokens, and roughly 20% API savings. Figure 2 illustrates cosine hubs versus an anisotropic ellipse; it is explanatory art, not an empirical embedding visualization. Figure 3 sweeps `α` and shows balanced Fusion best overall.

The privacy claim needs qualification. Unretrieved history remains local, but retrieved compressed text is transmitted to a commercial cloud service. Entity boosts can preferentially retain names, dates, and identifiers. Privacy is reduced exposure, not “preserved by default.”

## 4. Experimental design

LOCOMO contributes 1,542 QA pairs across single-hop, multi-hop, temporal, open-domain, and two adversarial questions. LongMemEval-S contributes 266. Metrics are ROUGE-L, Hit@k, and a strict GPT-4o-mini yes/no judge. The same provider/model family participates in answer generation and judging, raising correlated-bias risk.

Baselines include NaiveRAG, Mem0, MemoryBank, MemoryOS, LLMLingua2, and SelectiveContext. The claim that Mem0 and MemoryOS differ only in “superficial” heuristics is dismissive and not necessary for the comparison. Their near-identical numbers do not prove retrieval metric is the dominant bottleneck.

Experiments run on an RTX 4080 Laptop GPU with 12 GB VRAM, while the deployment narrative emphasizes 8 GB. A later paragraph says profiling on an 8 GB setup but does not name the device. It reports a 3.2 GB combined encoder/distiller peak, then elsewhere says gte-Qwen2 alone uses about 5.9 GB. These VRAM statements cannot all describe the same configuration.

The paper reports one McNemar significance marker for the strongest LOCOMO result. It does not report confidence intervals, repeated API runs, judge reliability against humans, multiple-comparison correction, or cost variance.

## 5. Results

### 5.1 End-to-end retrieval

On LOCOMO with MiniLM, Fusion reaches Judge 0.540, ROUGE-L 0.227, Hit@10 0.532, 2.09M prompt tokens, and 2,232 seconds. NaiveRAG is 0.503/0.215/0.515, 2.05M tokens, and 2,904 seconds. Fusion is more accurate but sends slightly more prompt tokens, so this row does not demonstrate compression savings.

With gte-Qwen2, Fusion Judge 0.536 barely exceeds NaiveRAG 0.531 and trails it on ROUGE-L and Hit@10. The discussion correctly says covariance correction helps smaller embeddings more.

LongMemEval-S MiniLM Judge is 0.462 for Fusion versus 0.444 Mem0 and 0.436 NaiveRAG. Time is 659 seconds versus 632 for CoreMem-V3 and 712 for NaiveRAG. The paper also reports a smaller +0.75-point gain on LongMemEval-M.

### 5.2 Compression stress test

Under a fixed NaiveRAG retriever and budget 1,000, SmolLM2 FDTD has conditional accuracy 0.9756, agreement 0.9688, and compressed/original ratio 67.4%. Yet the appendix judge table at the same budget shows truncation 0.4583 versus SmolLM2 0.4271. Conditional accuracy excludes or conditions on a subset and cannot substitute for overall answer accuracy. The manuscript’s “highest” claim depends on metric conditioning.

At budgets 500 and 250, simple truncation beats every learned compressor on Judge accuracy. This is one of the most important findings and limits the “strict budget” positioning.

### 5.3 Structural ablations

The structural table is internally difficult to interpret. At budget 500, removing content boost raises Hit@1/Hit@10 from 0.0208/0.0833 to 0.0938/0.375 while lowering ROUGE-L. At budget 1,000, removing gap fill collapses Hit metrics. Retrieval is supposedly fixed in the compression-only test, so “Hit” must be defined for compressed evidence, not retriever recall; that definition is absent.

The prose claims removal of syntax/keyword boost causes a 4.5-point Judge collapse, but the displayed ablation table contains only ROUGE-L and Hit metrics. Full judge results must be released.

### 5.4 Case studies and profiling

The appendix includes six chosen retrieval successes, five compression successes, and one balanced failure where Fusion retrieves car memories instead of photography. Selection criteria are qualitative and cannot estimate frequency.

Reported edge compression takes about one second, while cloud completion takes 140–155 seconds. That comparison reflects network/provider latency and makes almost any local preprocessing look negligible. A local or faster-cloud deployment could change the trade-off.

## 6. Ablations

The paper varies embedding model, fusion weight, distiller model, compression budget, and structural masks. It does not conduct the authors’ acknowledged full factorial interaction of retrieval and compression, so the end-to-end contribution of FDTD is unclear. CoreMem end-to-end prompt tokens are near the uncompressed baselines, suggesting the main table primarily tests retrieval.

Pure inverse-covariance retrieval helps temporal/open questions but can hurt multi-hop bridge facts. Fusion is a sensible hedge, yet `α=0.5` is chosen on the evaluation family and needs a held-out calibration protocol.

## 7. External context

## Source and evidence notes

The arXiv record confirms v1, all seven authors, and the two-benchmark headline.[^1] LOCOMO and LongMemEval primary records establish the benchmark definitions; they do not validate CoreMem’s implementation.[^2] The source names an anonymous 4open.science code/data snapshot, but direct access was blocked in the review environment and no searchable durable repository was verified.[^3]

The paper declares Gemini for language/TeX polishing and GPT Image 2 for figure assistance. It reports no funding or conflict statement. Full source documents were verified locally and remain withheld from public deposition.[^4]

No same-paper DEP-A, DEP-E, or relevant Black-Lake-Data record was found. Related memory records are conceptual rather than direct implementations.

## 8. Code, data, and reproducibility status

Reproduction is blocked until the authors pin executable code and clarify which covariance algebra, selection hierarchy, and score directions were used. Required artifacts include memory embeddings, exact benchmark subsets, answer/judge outputs, API versions, all seeds, human judge agreement, environment lockfiles, GPU traces, and definition/code for conditional accuracy, agreement, and compressed Hit metrics.

The code must include unit tests for matrix dimensions, equivalence with dense covariance inversion, budget preservation after gap fill, and α endpoint semantics.

## 9. Independent re-conceptualization

CoreMem should be reframed as **whiten–retrieve–explain–compress**:

- whiten embeddings with a validated shrinkage covariance estimator;
- retrieve with a held-out-calibrated blend of whitened and cosine scores;
- explain retrieval using source turns and per-dimension contributions; and
- compress with query-conditioned discrete selection under exact syntax and budget constraints.

The compressor should use a downstream relevance objective, such as contrastive answer evidence or a small query-conditioned scorer, rather than self-likelihood of the context. Structural constraints can be expressed as a dependency graph or span-selection integer program, avoiding ad hoc post-budget gap fill.

### A record-level cost model

The edge/cloud split should be evaluated over a complete memory lifecycle. Index construction requires embedding all memories and estimating or updating covariance. New turns change the mean, variances, low-rank directions, normalization extrema, and potentially every ranking. A practical system needs either periodic rebuilds or an online covariance update with versioned snapshots. Query cost should include embedding, candidate search, covariance scoring, local forward/backward saliency, tokenization, cloud transfer, and generation. Storage should include raw encrypted turns, embeddings, covariance factors, and audit logs.

This accounting could reverse conclusions at different scales. For a few hundred memories, exhaustive `O(Ndr)` scoring may be acceptable and hubness correction useful. At one hundred thousand memories, the paper already anticipates overhead; an approximate index that first retrieves a candidate set may dominate. The covariance transform can be folded into whitened embeddings and queried with maximum inner product search, but only if the matrix is fixed and positive definite. That is a more conventional and testable systems design than applying a bespoke score over every item.

### Evaluation without circular judges

The cloud generator and GPT-4o-mini judge share provider lineage and may share linguistic biases. A robust protocol should use human adjudication on a stratified subset, a second independent judge family, exact evidence citation, and answer normalization. It should publish disagreements rather than reducing everything to a binary prefix check. The two adversarial examples in LOCOMO are far too few for safety conclusions.

## 10. Implications

The data suggest covariance correction is most useful when embeddings are small and hub-prone. That is a useful deployment principle independent of the Fisher branding. It may extend inexpensive encoders without upgrading to multi-billion-parameter retrieval models.

The compression findings imply that “learned” pruning is not automatically robust at severe budgets. Simple truncation retains discourse coherence and can outperform saliency methods. Future work should report absolute downstream accuracy, not only conditional preservation among already-correct examples.

## 11. Research notes, limitations, and deployment applicability

Use is plausible for local personal assistants when users accept selected-history transmission to a cloud model. Production needs explicit consent, redaction, audit logs, encryption, retention policy, and a local-only option. Entity boosts increase both factuality and privacy exposure.

The authors acknowledge English-only evaluation, scaling beyond 100,000 memories, flat storage, edge-model limits, and missing factorial ablation. Further limits include invalid formal claims, ambiguous hardware, one provider/judge, no streaming update cost, no adversarial privacy test, and no comparison with modern approximate nearest-neighbor whitening baselines.

## 12. Falsifiable hypotheses

1. Correct feature-space Woodbury retrieval should match dense inverse-covariance rankings to numerical tolerance; the printed pseudocode should fail this test.
2. Shrinkage covariance with held-out λ should outperform the fixed `10×mean` rule on hubness and accuracy.
3. Query-conditioned span selection should beat self-likelihood FDTD at budgets 250–1,000.
4. Exact syntax-constrained selection should maintain budget and outperform post-hoc gap fill.
5. Fusion gains should shrink with stronger encoders and grow with measured hubness.

## Replication and falsification agenda

Reproduce LOCOMO and LongMemEval with released outputs, three API repetitions, blinded human judging, and paired intervals. Separate retrieval-only, compression-only, and full factorial experiments. Compare cosine, centered cosine, whitening, Mahalanobis, local covariance, mutual proximity, and approximate nearest-neighbor baselines. Report hub occurrence, recall, latency, memory, build/update cost, and answer quality.

For FDTD, compare gradient-times-input, squared gradient/Fisher estimates, attention, surprisal, query relevance, span classifiers, and truncation. Enforce identical retained-token counts. Test deletion perturbations directly and withdraw the theorem unless a valid bound covers discrete position shifts and the actual cloud loss.

## 13. Durable restatement

CoreMem reports that a blend of cosine and covariance-whitened retrieval can improve small-embedding dialogue memory, and that syntax-aware discrete pruning can preserve useful text around a 1,000-token budget. Its empirical decomposition and failure reporting are valuable.

The method is not presently a validated Fisher–Rao system, its Woodbury algorithm is malformed, and its KL theorem is unsupported. The public takeaway should be empirical: covariance-aware retrieval plus inspectable text compression is promising on edge devices, pending corrected mathematics, code, hardware evidence, and independent reproduction.

## Evidence boundary

- **Paper report:** all result values, hardware statements, and architectural descriptions attributed to the authors.
- **External primary evidence:** arXiv and benchmark primary records.
- **Reviewer inference:** matrix audit, theorem assessment, privacy and fairness analysis.
- **Hypothesis/proposal:** corrected covariance retrieval, exact span selection, and replication tests.
- **Not performed:** code execution, API calls, model downloads, or benchmark reproduction.
