# Learned Admission as the Write Path for Conversational Memory

## A detailed review, technical reconstruction, and independent re-conceptualization of “MemRouter: Memory-as-Embedding Routing for Long-Term Conversational Agents”

**Paper:** Tianyu Hu, Weikai Lin, Weizhi Zhang, Jing Ma, and Song Wang, *MemRouter: Memory-as-Embedding Routing for Long-Term Conversational Agents*, arXiv:2605.00356v1, 1 May 2026.[^1]  
**Canonical record:** https://arxiv.org/abs/2605.00356  
**Reviewed sources:** locally archived full PDF and full-paper HTML for v1; the canonical arXiv record; the paper-linked GitHub repository; the official LoCoMo paper and repository; and the canonical Memory-R1 record.[^2]  
**Review scope:** full paper, appendices, ten tables, four figures, nine numbered equations, limitations, code/data statements, and external primary-source checks.  
**Reviewer activity:** source reconstruction and claim vetting only. No training or benchmark experiment was rerun.

---

## Executive assessment

### The paper in one sentence

MemRouter replaces an autoregressive language-model call on every conversational turn with a small supervised classifier built around frozen embedding and transformer components, deciding whether a turn enters long-term memory while leaving later retrieval and answering to separate modules.

### The deeper idea

The work is best understood as a separation-of-concerns argument. A memory-enabled agent usually hides at least three policies behind the word “memory”: what to write, what to retrieve, and how to answer from retrieved evidence. MemRouter isolates the first. Its central bet is that admission is closer to contextual classification than generation. A frozen BGE encoder compresses chunks of recent dialogue, a learned projector maps those vectors into the hidden space of a frozen Qwen transformer, and lightweight heads predict an operation and content category. When the operation is ADD, the system stores the original turn verbatim rather than asking the model to generate a rewritten memory.

That design has two attractions. First, it removes decoding latency and generated-memory drift from the write path. Second, it makes the admission decision trainable with only about twelve million updated parameters. The paper reports that this router beats a locally recreated LLM memory manager under a controlled LoCoMo harness while reducing write-management latency substantially. The evidence also shows, however, that the overall system remains highly dependent on retrieval design and category-specific answer prompting. MemRouter is not a complete memory architecture and should not be read as one.

### Bottom-line judgment

This is a useful and well-targeted systems contribution. The strongest result is the matched-harness comparison: with retrieval, prompts, and Qwen2.5-7B answering held fixed, the learned router reports 52.0 overall F1 versus 45.6 for an LLM manager, alongside p50 memory-management latency of 58 ms versus 970 ms. The operation is narrow, measurable, and plausibly deployable. The budget sweep further suggests that learned admission matters increasingly as memory capacity tightens.

The paper’s broadest claims need qualification. The strongest main-table baseline numbers are not all produced under the same local harness. A transfer experiment retrains router heads for the new backbone and, on Llama, scores below store-all overall; it therefore supports modularity more than plug-and-play backbone invariance. Category-specific prompts account for a large fraction of final performance, and open-domain questions remain weak. The public code URL named by the paper resolved to an empty repository at review time, so the implementation and reported timings were not independently auditable.[^3]

### Principal strengths

- It identifies write admission as a distinct bottleneck instead of conflating it with retrieval and answer generation.
- It uses a controlled matched harness for its most defensible causal comparison.
- It reports latency distributions, total processing time, QA throughput, and peak GPU memory rather than a single accuracy number.
- It includes useful negative evidence: GRPO-only collapse, UPDATE overprediction, and the limited gain from retrieval refinements relative to prompting.
- It exposes storage-budget curves and policy comparisons rather than evaluating only at one retained-memory ratio.
- It stores original text, preserving an audit trail and avoiding semantic distortion by a generated summarizer.

### Principal qualifications

- The router is trained from labels generated by a much larger teacher, moving rather than eliminating some cost.
- The best policy uses only ADD and NOOP; richer update semantics failed in the reported setup.
- LoCoMo contains only ten conversations, so nested QA items do not create ten-independent-conversation-scale certainty.
- Imported baseline numbers and locally controlled numbers must not be pooled as if they shared one evaluator.
- Retrieval and answer prompts materially influence the final score; admission alone does not explain the main result.
- The paper-linked code repository was empty, preventing code/default/hardware reproduction checks.

## 1. The problem the paper is actually solving

### 1.1 Write-path cost in long-running agents

A persistent conversational agent accumulates far more turns than should remain in an active prompt. External memory is one response, but external memory creates a write-control problem: storing everything increases retrieval noise and cost; storing too little destroys evidence. Many systems ask an LLM to interpret every turn and emit ADD, UPDATE, DELETE, or NOOP, sometimes with a rewritten memory. This can be flexible, but it adds autoregressive decoding to every interaction. It may also produce lossy or fabricated restatements before the system ever attempts retrieval.

MemRouter narrows the problem to a binary decision: should the current conversational content be retained? The proposed system must account for context because the importance of “Yes, next Wednesday” is invisible without preceding turns. Yet it should decide more cheaply than a generative manager and avoid changing the source text.

### 1.2 Prior families and the claimed gap

The paper distinguishes heuristic storage, generative memory management, and learned memory agents. Store-all is simple but budget-insensitive. Keyword and recency rules are cheap but brittle. Generative managers can synthesize structured memory, while Memory-R1 learns memory operations with reinforcement learning. MemRouter claims an intermediate point: contextual reasoning through frozen pretrained components, supervised classification rather than generation, and a verbatim store.

This gap is credible. A write policy does not inherently need to generate language. But the reduction changes the capability boundary. Binary admission cannot merge duplicates, correct stale facts, or represent belief revision. The paper’s own attempt to expand the operation space supplies evidence that this is not just an omitted feature: the UPDATE option proved difficult to learn and caused overprediction.

### 1.3 What novelty belongs to MemRouter

The inherited components are substantial: BGE supplies turn embeddings; Qwen supplies contextual transformation; BM25 and dense retrieval supply recall; LoCoMo supplies data and scoring; a larger Qwen teacher supplies labels. The novelty is their arrangement into a low-parameter, no-decoding write router, plus the experimental isolation of admission policy. The contribution is architectural and empirical, not a new embedding objective, transformer, or retrieval algorithm.

## 2. Formal and technical reconstruction

### 2.1 Dialogue chunking (Equation 1)

Let the current conversation history be divided into chunks, with no more than thirteen chunks and no more than five turns per chunk. In plain language, the router does not feed a long token transcript into the trainable module. It creates a short sequence of chunk-level representations covering at most the configured recent history. This bounds cost but also imposes a history horizon. The formula is an indexing definition, not a guarantee that all earlier dependencies remain recoverable.

### 2.2 Embedding and projection (Equations 2–3)

Each chunk is encoded using BGE-large-en-v1.5, yielding a 1,024-dimensional vector. A learned two-layer projector maps that vector into the 3,584-dimensional hidden space expected by Qwen2.5-7B. If the BGE representation is `e_i` and the projector is `P`, the mapped vector is `z_i = P(e_i)`. The projector contains roughly eight million trainable parameters.

This interface is the key engineering bridge. The Qwen transformer normally receives token embeddings distributed according to its learned vocabulary and embedding table. MemRouter supplies projected sentence/chunk embeddings instead. Training the projector must therefore learn both dimensional translation and a usable pseudo-token distribution. The paper demonstrates task performance but does not directly measure whether projected points occupy a stable or interpretable region of Qwen hidden space.

### 2.3 Frozen contextual transformer (Equation 4)

The projected chunk sequence is passed through the frozen 28-layer Qwen2.5-7B transformer. The final contextual representation of the relevant chunk becomes the feature for classification. Because the transformer is frozen, gradients update the projector and heads but not Qwen. “Frozen” lowers trainable-parameter count; it does not mean the 7B backbone disappears from inference memory or computation. This distinction explains why the router can be parameter-efficient yet still reach a higher reported peak GPU footprint than the generative manager in the paper’s particular setup.

### 2.4 Classification heads and loss (Equations 5–6)

One head predicts the operation, ultimately ADD or NOOP. A second predicts a content type: key fact, emotional content, preference, plan, or routine. Both heads are small learned classifiers, approximately four million parameters together. The loss is a weighted sum of operation cross-entropy and type cross-entropy. The type term is masked to ADD examples, since assigning a retained-memory type to a discarded turn would be semantically meaningless.

The signs and normalization are conventional: negative log-likelihood is minimized, and the type loss is conditioned on the ADD label. The conditioning matters. An implementation that averaged type loss over all samples or encoded NOOP as a sixth type would optimize a different problem. The paper does not claim calibrated probabilities, so thresholds should not be interpreted as confidence estimates without additional calibration.

### 2.5 Verbatim memory and retrieval (Equations 7–9)

For an ADD prediction, the system stores a tuple containing timestamp, speaker, and original text. No generative summary is introduced. At question time, it retrieves up to sixty items with a hybrid score combining dense BGE similarity and BM25, using a dense weight of 0.7. Speaker and temporal adjustments can boost scores, and a session-diversity mechanism avoids returning only redundant neighboring items.

The final answer model receives retrieved memories under question-category-specific prompts. This is outside the router but essential to reported end-to-end F1. Equation 8’s convex mixture assumes dense and lexical scores have been normalized to commensurate scales; the paper specifies the mixture but does not establish that the coefficient transfers to other corpora. Equation 9 adds heuristic boosts, making retrieval a configured subsystem rather than a neutral readout of admission quality.

## 3. Architecture and information flow

### 3.1 Training plane

The label-production plane begins with turns drawn from LoCoMo, LongMemEval, and MSC. Qwen3.5-35B-A3B acts as a teacher that assigns storage operations and types. The authors report 28,415 labeled turns. The student path chunks contextual history, computes BGE vectors, projects them into Qwen hidden space, processes them with frozen Qwen2.5-7B, and trains only the projector and heads for five epochs.

This is knowledge distillation in a broad operational sense, though the targets are hard action labels rather than teacher logits. The teacher’s preferences become the supervision boundary. If the teacher systematically ignores subtle future-relevant details, ordinary train/test separation does not remove that policy bias.

### 3.2 Online write plane

At each eligible turn, the same embedding–projection–frozen-transformer pipeline produces an operation. NOOP stops processing. ADD commits the original tuple and its predicted type. Because there is no token decoding, latency depends mainly on encoder and transformer forward passes rather than output length. The system still requires both an embedding model and a 7B transformer, so “lightweight” describes tuned parameters and the absence of decoding more accurately than total resident model size.

### 3.3 Read and answer plane

When a user question arrives, dense and lexical search score stored items. Optional speaker, time, and diversity logic reshape the ranking. Retrieved items are passed to an answer model, Qwen2.5-7B in the controlled experiments and Qwen3.5-35B in a scaling result. Prompts differ by question category. End-to-end accuracy therefore reflects four coupled choices: admission, retained text, retrieval, and answer prompting/model.

### 3.4 Dependency map

The router depends on teacher-labeled data, BGE behavior, a compatible frozen transformer, projector training, and a class distribution that does not collapse. The full agent additionally depends on a lexical index, dense index, metadata fields, question categorization, and answer prompts. Operational failures can occur at any boundary. A correct ADD decision is not useful if retrieval misses the entry; a correct retrieval is not useful if answer prompting mishandles temporal reasoning. The small trainable delta should not be mistaken for the whole resident system.[^6]

## 4. Complete operational pipeline

1. Partition recent dialogue into at most thirteen chunks of at most five turns.
2. Encode every chunk with BGE-large-en-v1.5.
3. Map each 1,024-dimensional vector through the learned projector into 3,584 dimensions.
4. Contextualize the mapped sequence with frozen Qwen2.5-7B transformer layers.
5. Predict ADD/NOOP and, for ADD, one of five memory types.
6. Store the original timestamp, speaker, and text for ADD decisions.
7. At query time, compute dense and BM25 scores over stored items.
8. Mix scores at the configured 0.7 dense weight and apply metadata/diversity adjustments.
9. Select up to sixty memories.
10. Assemble a category-specific prompt and ask the downstream model for an answer.
11. Score the answer with LoCoMo’s category-sensitive protocol.

The atomic deployment unit is not merely the twelve-million-parameter delta. It includes frozen BGE, frozen Qwen, trained weights, indexes, metadata normalization, retrieval coefficients, question routing, prompts, and answer-model configuration.

## 5. Independent re-conceptualization

### 5.1 Admission control as a learned lossy channel

MemRouter can be reframed as a controller on a lossy channel between a high-rate event stream and a bounded evidence store. ADD transmits an item without semantic rewriting; NOOP discards it irreversibly. Under this view, admission error is asymmetric. A false positive consumes capacity and can add retrieval noise. A false negative destroys evidence before any future query is known. The optimal threshold therefore depends on storage price, retrieval robustness, and the unknown distribution of future questions—not only label accuracy.

### 5.2 A compiler front end for memory

The architecture also resembles a compiler front end. BGE produces a compact intermediate representation; the projector adapts the representation to a frozen reasoning substrate; Qwen contextualizes the sequence; the heads emit a small instruction set. The failed UPDATE experiments suggest that a richer instruction set needs stronger state semantics, not simply more output classes. An UPDATE action must identify a target, decide replacement versus merge, and preserve temporal provenance.

### 5.3 Decoupled write policies as reusable infrastructure

The durable principle is to evaluate write policy separately from read policy. A team could compare learned admission, rules, store-all, and generative managers against the same downstream index and answer harness. This creates a clean systems boundary and prevents an improved prompt from being credited to memory selection. MemRouter’s paper is valuable partly because several appendix experiments move toward this decomposition.

## 6. Experimental design reconstructed

### 6.1 Data and splits

The principal evaluation uses the official ten-conversation LoCoMo release.[^4] The paper reports a conversation-level 1:1:8 train/validation/test split and excludes test turns from teacher labeling and tuning. It evaluates 1,307 non-adversarial QA items. The authors describe conversations as roughly 35 sessions, 588.2 turns, and 16,618.1 tokens on average in their processed setup. Teacher labels also use LongMemEval and MSC turns, broadening supervision beyond the evaluation corpus.

Conversation-level splitting is the right direction because turn-level random splits would leak personas and recurring events. But eight held-out conversations remain a small number of independent dialogue clusters. Confidence intervals over questions must account for nesting to justify population-level precision.

### 6.2 Training configuration

The reported configuration trains for five epochs with batch size 16 and learning rate `1e-3` on one NVIDIA B200. Router training takes about ten minutes. Teacher labeling runs at 4.8 turns per second and takes about 98 minutes. These numbers describe the authors’ environment, not a hardware-normalized cost model. They also omit the cost of pretraining the inherited models, appropriately outside the experiment but relevant to total system footprint.

### 6.3 Baselines and controls

Main-table baselines include RAG, A-Mem, Mem0, MemoryOS, and Memory-R1-GRPO. Several are values reported by prior work. The more causally useful Table 4 recreates an LLM-based memory manager under matched retrieval, answer prompts, and Qwen2.5-7B answering. Storage-policy comparisons include store-all, random, recent-k, keyword, and a shallow MLP. Ablations vary dense encoder, BM25, metadata grouping, category prompts, operation space, supervision regime, answer model, and memory budget.

### 6.4 Metrics and denominator semantics

The main metric is token-level F1 with special evaluation rules for categories such as temporal and open-domain questions. Overall F1 aggregates single-hop, multi-hop, temporal, and open-domain subsets. Because prompt templates change by category, the metric measures a configured task pipeline. It should not be read as a universal memory accuracy percentage.

### 6.5 Runtime semantics

The efficiency study covers roughly eight conversations, about 5,000 turns, and 1,300 questions. It reports per-turn management p50/p95, total management time, QA p50, QA throughput, and peak GPU memory. These are useful but topology-specific: batching, model residency, quantization, index size, and concurrency can change the ranking. The reported system uses no write-side decoding, so its latency advantage should be directionally robust even if exact multipliers are not.

## 7. Results and their proper interpretation

### 7.1 Main LoCoMo results (Table 1)

The reported overall F1 values are RAG 9.0, A-Mem 26.1, Mem0 30.6, MemoryOS 34.6, Memory-R1-GRPO 43.1, MemRouter with a 7B answer model 52.0, and MemRouter with a 35B answer model 55.5. For the 7B system, category scores are 57.5 single-hop, 52.4 multi-hop, 44.0 temporal, and 27.1 open-domain. With the 35B answer model they become 60.9, 56.1, 49.7, and 23.8.

The 8.9-point difference over Memory-R1-GRPO is headline-worthy but not a controlled single-factor effect because prior-system numbers can differ in harness details. The 35B result demonstrates answer-model sensitivity: three categories improve, while open-domain falls by 3.3 points. Larger generation capacity is therefore not monotonically better across this evaluator.

### 7.2 Backbone transfer (Table 2)

With Llama as answer backbone, store-all scores 50.6 and MemRouter 49.1; with the Qwen setup, MemRouter scores 52.0. Router heads are retrained for the Llama experiment. This supports the claim that the architecture can be adapted across backbones. It does not show that one trained router transfers unchanged, and it does not show an overall advantage over store-all for Llama. Temporal performance is the notable area where learned selection helps.

### 7.3 Matched storage budget (Table 3)

At a roughly 62% storage ratio, overall F1 is 53.7 for store-all, 50.8 for MemRouter, 50.0 for the MLP, 47.2 for keyword selection, 42.8 for random, and 38.6 for recent-k. Store-all remains best when it is allowed to retain everything; MemRouter is the best constrained learned/heuristic policy in this comparison. The distinction is important: the contribution is efficient selection under limits, not evidence that discarding information inherently improves every unconstrained setting.

### 7.4 Controlled manager comparison (Table 4)

Under the matched harness, the LLM manager reports 45.6 overall F1 with an interval of [42, 49], versus MemRouter’s 52.0 with [51, 53]. Category results are 50.4 versus 57.5, 44.2 versus 52.4, 43.8 versus 44.0, and 20.5 versus 27.1. The near tie on temporal questions suggests the aggregate improvement is concentrated elsewhere. The non-overlapping reported intervals are supportive, but the paper’s inferential strength depends on how resampling respects conversation clusters.

### 7.5 Efficiency (Table 5)

Memory-management p50 latency falls from 970 to 58 ms and p95 from 1,415 to 71 ms. Total write-management time falls from 5,028 to 293 seconds. QA p50 falls from 198 to 141 ms, and throughput rises from 0.24 to 2.58 QA/s. Peak GPU memory, however, rises from 26.6 to 31.2 GB. The correct summary is therefore “lower latency and higher throughput in the tested stack, with higher measured peak GPU memory,” not an unqualified resource reduction.

### 7.6 Budget curve (Table 6 and budget figure)

At store ratios 94.5%, 80.4%, 64.9%, 45.9%, and 19.4%, MemRouter scores 53.7, 53.2, 52.0, 46.1, and 33.5. Random selection scores 52.7, 47.4, 41.8, 34.8, and 23.3. The gaps are 1.0, 5.8, 10.2, 11.4, and 10.2. Learned admission has little value near store-all capacity but clear value under moderate and severe constraints. Performance still degrades sharply at the tightest budget, showing selection cannot substitute for missing evidence indefinitely.

## 8. Ablations and causal evidence

### 8.1 Retrieval and prompting (Table 7)

MiniLM dense retrieval gives category scores 29.5/18.9/11.0/13.9. BGE improves them to 44.1/33.8/19.4/12.8. Adding BM25 yields 44.5/35.4/19.9/14.2. A speaker boost changes almost nothing. Speaker-grouped formatting reaches 54.8/26.1/34.5/17.8, trading multi-hop quality for single-hop and temporal gains. Category-specific prompts produce the final 57.5/52.4/44.0/27.1.

This table is one of the paper’s most important. It shows that the answer interface is not incidental. Large gains arrive after representation and formatting changes downstream of admission. Any reproduction that reports only the router checkpoint without the exact prompt/evaluator stack is incomplete.

### 8.2 Factor attribution (Table 8 and factor figure)

Descriptive averages put admission policies at 38.7 random, 47.9 MLP, 48.3 keyword, 49.0 MemRouter, and 50.5 store-all. Generic prompting averages 44.3 versus 49.5 for category prompting. Cosine retrieval averages 46.5 versus 47.2 for hybrid retrieval. These are factorial summaries, not necessarily unbiased causal effects from a fully randomized experiment. They nevertheless reinforce a practical ordering: admission and prompts matter more than the reported dense-plus-BM25 increment.

### 8.3 Supervision and operation space (Table 9)

GRPO-only training collapses to all NOOP and scores 1.1. Supervised initialization followed by GRPO reaches 28.5. A supervised ADD/UPDATE/NOOP system scores between 1.4 and 12.8 because UPDATE is overpredicted. Plain supervised ADD/NOOP reaches 52.0. This negative result constrains the method’s scope and suggests action semantics, reward design, or class balance are not yet adequate for stateful updates.

### 8.4 Answer-model scale (Table 10)

The 35B answer model raises overall F1 from 52.0 to 55.5, but open-domain F1 decreases. The router does not erase answer-model behavior, and model scaling should not be credited to memory admission. A future evaluation should cross admission policy, retriever, prompt family, and answer model with identical budgets.

## 10. External primary-source vetting

### 10.1 Publication and identity

The canonical arXiv record confirms the title, five authors, cs.CL/cs.AI classification, v1 submission date, PDF/HTML availability, and arXiv-issued DOI.[^1] No peer-reviewed venue was identified in that record at inspection. The review therefore treats this as an arXiv preprint, not a venue-accepted result.

### 10.2 LoCoMo provenance

The official LoCoMo paper describes a machine-human pipeline, persona and temporal-event grounding, human verification, and multi-session conversations; the official repository releases ten conversations and task annotations.[^4] This corroborates the benchmark’s provenance. It also reveals why conversation-level uncertainty is important: many QA items are nested inside a small set of dialogue worlds.

### 10.3 Memory-R1 context

The canonical Memory-R1 record describes a reinforcement-learning framework with manager and answer agents and operations ADD, UPDATE, DELETE, and NOOP across LoCoMo, MSC, and LongMemEval.[^5] It is a relevant predecessor and baseline, but its latest record is v5 whereas imported scores may correspond to a specific configuration/version. Comparisons should retain that version and harness boundary.

### 10.4 Code and data availability

The paper-linked GitHub URL existed as a public repository but displayed no files or commits usable for reproduction at inspection.[^3] The official LoCoMo data/code repository was available.[^4] No independent MemRouter checkpoint, training script, prompt pack, or environment lockfile was verified. Consequently, this review can audit the paper’s specification but cannot compare implementation defaults with prose.

### 10.5 Associated DEP search

Exact arXiv ID and normalized-title searches across the live Black Lake DEP-A and DEP-E indexes, DEP READMEs, and the relevant Black-Lake-Data deposition tree found no same-paper or directly corresponding implementation record. Conceptual proximity alone was not treated as an association.

## 11. Research notes and critical considerations

### 11.1 Internal consistency

The central architecture, parameter counts, and ADD/NOOP result are coherent across the main text and appendices. The phrase “backbone-agnostic” is looser than the table evidence because heads are retrained. Efficiency language should separate trainable parameters, latency, throughput, and peak GPU memory. The paper does report the higher peak, which makes the necessary correction possible.

### 11.2 Label-policy dependence

The router learns a teacher’s admission decisions, not future utility directly. A turn’s usefulness is question-dependent and can emerge long after storage. Teacher labels are a practical shortcut, but they may encode verbosity, salience, or taxonomy biases. The budget curve demonstrates that the policy outperforms random selection, yet it does not reveal calibration or the distribution of false negatives by future question type.

### 11.3 Metric sensitivity

Category-specific prompts materially change F1. The evaluator’s token overlap and special-case rules may reward answer form as well as evidence access. A stronger study would report retrieval recall to gold supporting turns, admission recall for later-used turns, answer accuracy conditional on successful retrieval, and cluster-bootstrap intervals by conversation.

### 11.4 Cost accounting

Online write latency is the paper’s strongest operational metric. Full accounting should also include teacher-label generation, embedding index construction, model load/residency, and maintenance as user facts change. Because the system stores verbatim text, it avoids summary-generation cost but can retain redundancy and personally sensitive text unless policy controls intervene.

### 11.5 Alternative explanations

MemRouter’s advantage over the recreated LLM manager could arise from more consistent binary decisions, better alignment of the teacher labels with LoCoMo, avoidance of bad generated summaries, or differences in storage ratio—not only embedding routing. The controlled harness narrows the alternatives but a matched admission-recall analysis would distinguish them.

### 11.6 Unresolved questions

- How stable are thresholds under different ADD base rates or dialogue domains?
- Does the pseudo-token projector remain stable under a changed embedding encoder?
- Which future-relevant facts become false negatives, especially in open-domain questions?
- Can a richer action system work if UPDATE is decomposed into target selection and edit type?
- Are the reported intervals clustered by conversation, session, or question?
- How much of the throughput gain persists when multiple online users share already-resident models?
- Can the router be distilled further without the frozen 7B forward pass?

## 12. Limitations, implications, and applicability

### 12.1 Limitations stated or implied by the paper

The authors acknowledge weak open-domain performance, the restricted ADD/NOOP action space, the failure of UPDATE, and teacher-label cost. The evaluation is centered on LoCoMo and a specific retrieval/prompt stack. Code unavailability adds a present reproducibility limitation beyond the experimental limitations stated in the paper.

### 12.2 Scientific implications

The results support treating memory admission as its own learned task. They also show why “memory system” leaderboards are difficult to interpret: a prompt change can rival or exceed a retrieval refinement, and an answer model can change category behavior. Factorized evaluations should become standard.

### 12.3 Systems implications

For latency-sensitive agents, removing per-turn decoding from the write path is attractive. Verbatim storage simplifies provenance and debugging. The design is most applicable where immutable event logging is acceptable and later retrieval can absorb some redundancy. It is less suitable where memories require consolidation, deletion, legal retention controls, conflict resolution, or strong privacy minimization.

### 12.4 Governance implications

Automated admission is also automated retention. A high-recall policy may preserve secrets, health information, or transient emotional statements. Deployment needs explicit consent, tenant isolation, encryption, retention limits, deletion propagation, and an audit trail showing why an item was stored. Predicted “type” is not a sensitivity classification and should never substitute for data-loss-prevention controls.

## 13. New falsifiable hypotheses

### Hypothesis 1: utility-calibrated thresholds dominate fixed classification thresholds

**Derivation:** the cost of a false negative rises as the memory budget tightens and varies by question type.  
**Test:** calibrate ADD probabilities, then choose thresholds using a utility function combining storage cost and downstream evidence recall. Compare with the paper’s fixed policy across the full budget curve.  
**Prediction:** utility calibration improves area under the F1-versus-storage curve, especially below 50% retention.  
**Falsifier:** no improvement over fixed thresholds across held-out conversations and two additional dialogue domains.

### Hypothesis 2: direct future-use supervision beats teacher salience labels

**Derivation:** teacher labels approximate importance, while benchmark annotations reveal which turns later support answers.  
**Test:** train identical routers on teacher actions, future-use labels, and a mixed objective; report admission recall for gold evidence and end-to-end F1.  
**Prediction:** mixed supervision reduces false negatives and improves multi-hop and temporal scores without requiring store-all ratios.  
**Falsifier:** teacher-only labels match or exceed mixed supervision under controlled budgets.

### Hypothesis 3: decomposed UPDATE avoids operation collapse

**Derivation:** UPDATE conflates target retrieval, conflict detection, and editing.  
**Test:** split UPDATE into “link-to-prior-memory” classification plus a deterministic append/supersede record, retaining both versions.  
**Prediction:** the decomposed system avoids UPDATE overprediction and improves temporal contradiction handling.  
**Falsifier:** it still collapses or fails to beat ADD/NOOP on contradiction-focused tests.

### Hypothesis 4: a smaller contextual encoder preserves most of the latency benefit

**Derivation:** the 7B backbone is frozen and used for contextual classification, not generation.  
**Test:** distill its hidden decision function into 300M, 1B, and 3B encoders while preserving BGE/projector inputs.  
**Prediction:** a sub-1B model retains at least 95% of admission utility with lower peak GPU memory.  
**Falsifier:** all smaller models lose more than five F1 points at matched storage ratios.

## 14. Deployment and governance considerations

### 14.1 Appropriate use

The design is appropriate as a replaceable write-policy component in an agent platform with clear memory budgets, downstream retrieval, and a human-auditable event store. It is especially plausible for customer-support continuity, personal assistants with explicit consent, and internal copilots where latency at every turn matters.

### 14.2 Poor fit and failure modes

It is a poor fit where facts change frequently and supersession is mandatory, where a false negative is safety-critical, or where raw text cannot legally be retained. Failure modes include silent NOOP on future-critical details, duplicate retention, category misclassification, embedding drift, retrieval score miscalibration, prompt-router errors, and answer hallucination despite correct memory.

### 14.3 Required safeguards

- Preserve operation probabilities and reason codes for audit without presenting them as calibrated certainty.
- Apply independent sensitivity and retention policy before any write.
- Support user-visible memory inspection, correction, and deletion.
- Version embeddings, projectors, prompts, and index settings.
- Monitor retention rate, gold-evidence recall on canaries, retrieval miss rate, and category drift.
- Keep a store-all shadow log only in consented, short-lived evaluation environments.
- Fail conservatively when models or indexes are unavailable; do not silently claim durable retention.

## 15. Replication and falsification agenda

### 15.1 Minimum reproducible release

Release exact conversation IDs for the 1:1:8 split, teacher prompts and raw labels, operation/type class counts, projector/head definitions, optimizer state, seeds, checkpoint hashes, retrieval normalization, BM25 tokenizer, metadata boosts, category router, answer prompts, and evaluator version. The empty public repository currently supplies none of these implementation checks.

### 15.2 Reproduce the causal comparison

Run MemRouter, the LLM manager, store-all, random, recent-k, keyword, and MLP under identical storage budgets. Use the same retained text format, index, retrieval depth, answer model, prompts, and evaluator. Report results per conversation with cluster-bootstrap intervals and paired differences.

### 15.3 Decompose end-to-end error

Measure operation precision/recall, recall of turns later used as gold evidence, retrieval recall at 1/5/20/60, answer accuracy conditional on evidence retrieval, storage ratio, duplicate rate, and stale-fact rate. This turns an aggregate F1 gain into a mechanistic diagnosis.

### 15.4 Reproduce efficiency

Measure cold and warm model starts, batch size one and concurrent loads, model residency, index size, p50/p95/p99, energy, peak allocated and resident GPU memory, teacher-label amortization, and total cost per thousand turns. Compare quantized and smaller contextual backbones.

### 15.5 Stress generalization

Use LongMemEval, MSC, contradiction-rich dialogues, code-switching, low-resource languages, noisy speech transcripts, and privacy-sensitive synthetic cases. Freeze the router for a true zero-shot transfer test, then separately permit head retraining to measure adaptation.

## 16. Durable restatement

MemRouter’s lasting contribution is the proposition that conversational memory writing can be a contextual classification service rather than a generation service. It embeds bounded dialogue chunks, adapts them into a frozen transformer, and trains small heads to retain or discard the current content while storing accepted text verbatim. In the authors’ LoCoMo setup, this policy is faster than a generative manager and more effective under constrained budgets than simple selection rules. The evidence does not establish a complete, universally backbone-agnostic memory system: retrieval, prompts, answer model, teacher labels, and operation semantics remain decisive. The result is most valuable as a clean systems boundary and a strong invitation to evaluate admission, retrieval, and answering separately.
