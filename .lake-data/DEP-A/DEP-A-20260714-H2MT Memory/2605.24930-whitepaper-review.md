# Hierarchical Routing Is the Contribution; Reproducibility Is the Missing Layer

## A full-paper review and independent re-conceptualization of “H²MT: Semantic Hierarchy-Aware Hierarchical Memory Transformer”

**Paper:** Maryam Haghifam, Zifan He, Jason Cong, and Yizhou Sun, arXiv:2605.24930v1, 24 May 2026.[^1]  
**Scope:** complete 11-page PDF and full-paper HTML, TeX source, all twelve tables, all four figures, the central equation families, appendices, limitations, implementation notes, and external primary-source vetting.  
**Review activity:** conceptual reconstruction, equation and metric audit, table/prose cross-checking, and provenance review. No training run, checkpoint, routing experiment, or latency measurement was independently reproduced.

---

## Executive assessment

### The paper in one sentence

H²MT presents a useful coarse-to-fine retrieval architecture that compresses every document-tree node into one reusable latent token, but the current paper leaves enough training, evaluation, and release details unresolved that its unusually broad quality and latency tables should be treated as promising author-reported evidence rather than reproducible findings.

### The durable contribution

The durable idea is not merely “hierarchical memory.” It is the separation of offline structure construction from online query routing. H²MT first maps a document to a rooted semantic tree, constructs one latent memory vector per node in post-order, and then routes a query from root to leaves with per-parent top-*k* selection. This gives the system an explicit latency–coverage control and makes a clear architectural claim: long-document question answering can condition generation on a small set of cached latent node memories instead of re-prefilling raw text or retrieved summaries for every query.

The paper also exposes an important systems trade-off. Text retrieval such as RAPTOR preserves words and therefore tends to retain lexical evidence, but it pays index-construction and online generation costs. H²MT is faster after indexing because each selected node contributes only one vector, but it is lossy twice: once when local text is compressed into a node memory and again when child memories are aggregated into a parent. The HotpotQA comparison makes this boundary unusually visible: the GMM hierarchy raises H²MT ROUGE-L from 30.37 to 46.64, while SBERT+RAPTOR reaches 56.39 but takes much longer for total generation.

### Bottom-line judgment

The paper is technically interesting and its main mechanism is coherent. The strongest reported result is the consistency of the latency reduction against HMT: for five of six backbones, mean TTFT over four datasets falls from seconds to hundreds of milliseconds. Quality is more mixed. H²MT usually beats HMT’s four-dataset average and performs especially well on NarrativeQA and QASPER, while HMT remains stronger on several HotpotQA or OpenROAD cells. The authors correctly acknowledge that weak or induced hierarchies can misroute evidence.

The evidence is nevertheless not yet white-box reproducible. The paper does not provide an official code or checkpoint URL, seed counts, confidence intervals, hardware configuration, batch and decoding settings, exact train/test sample counts, or a complete hyperparameter schedule. The implementation note says the backbone is frozen except for LoRA and lightweight H²MT modules, but the manuscript does not disclose LoRA rank, target modules, optimizer, learning rates, epochs, reconstruction-loss use, routing weights, temperatures, or per-dataset *k*. Those omissions are material because the tables compare many backbones and several baselines across latency, quality, memory, and privately constructed technical data.

There is also a publication-history qualification. OpenReview records an earlier H2MT submission to ICLR 2026 as withdrawn.[^2] That earlier version described a materially narrower evaluation and a somewhat different mechanism vocabulary. The arXiv manuscript is therefore best read as a later, expanded preprint rather than a conference-validated result. This is not evidence of error, but it matters for status and provenance.

The fairest conclusion is that H²MT supplies a strong architectural hypothesis and a useful experimental map. It does not yet supply the artifacts or statistical controls needed to establish the reported frontier. A release containing training code, hierarchy builders, datasets or redacted substitutes, exact configurations, checkpoints, and raw timing traces would substantially raise confidence.

## 1. Problem framing and scope

Long-context inference has at least three distinct costs. Dense attention grows rapidly with prompt length; prefill latency delays the first generated token; and the KV cache plus intermediate activations can exceed accelerator memory. Retrieval reduces input length, but a conventional retriever still returns text. Hierarchical RAG adds offline clustering and summary generation, and selected summaries or chunks must again be tokenized and prefixed to the generator.

H²MT asks whether a document’s native hierarchy can serve as the retrieval index and whether each retrievable unit can be stored as one model-space vector. Manuals, papers, and books naturally contain sections, subsections, chapters, scenes, and paragraphs. For unstructured collections, the authors use paragraph or chunk segmentation and, for HotpotQA, a Gaussian-mixture hierarchy over learned chunk memories.

The scope is document-grounded question answering, not general continual memory. The document is static enough that all node embeddings can be constructed and cached offline. Each query then pays for query encoding, tree routing, and generation from selected memories. This amortization assumption is essential. A rapidly changing corpus would require recomputing memories along every modified node’s ancestor path and could lose the advertised advantage.

The method keeps the backbone attention blocks unchanged, but “unchanged” should not be confused with training-free. H²MT inserts learned write/read embeddings, learns query and key projections, may learn an aggregation module, and applies LoRA adapters to the language model. Its contribution is a plug-in interface and training scheme, not a zero-shot retrieval wrapper.

## 2. Architecture and pipeline reconstruction

### 2.1 Semantic hierarchy

The document becomes a rooted tree `T=(V,E,r)`. Each node `v` has local text `x_v`; edges encode containment and scope. Leaves contain bounded text regions. Internal nodes can contain headings or local explanatory text plus children. Oversized units are recursively divided into paragraphs and then fixed-length chunks.

For technical manuals, table-of-contents metadata defines the hierarchy. NarrativeQA uses document → chapter or scene → paragraph. QASPER uses paper → section → paragraph. HotpotQA’s default construction treats each Wikipedia title as a root with sentence leaves, which is shallow and poorly aligned with multi-hop reasoning. A later experiment replaces it with recursive GMM clustering of node memories.

This preprocessing is simultaneously an inductive bias and a failure surface. A good hierarchy narrows search; a bad parent assignment can hide evidence behind an irrelevant ancestor. Per-parent top-*k* reduces early global collapse, but it cannot recover a branch that was pruned at a higher level.

### 2.2 Leaf memory construction

Each leaf receives one latent vector `m_v ∈ R^d`. The method inserts shared learnable write and read vectors around the leaf token embeddings:

`Ẽ_v = [e_write ; E_v ; e_read]`.

The causal Transformer processes this sequence, and the last-layer hidden state at `e_read` becomes `m_v`. Because the read position follows the text, it can attend to all prior positions. In plain language, the model learns to squeeze the bounded leaf content into a single state located at the read token.

The equation is dimensionally consistent: if `E_v` contains `n_v` token embeddings, the augmented sequence has `n_v+2` rows and hidden width `d`. What is not established is how much information one vector can retain as `n_v` grows, or whether rare exact strings survive. The later RAPTOR comparison suggests that the loss of surface detail is real.

### 2.3 Internal-node memory construction

Children are processed before their parent. Their memories are stacked in document order into `M_v^ch ∈ R^(c_v × d)`, where `c_v` is the number of children. An aggregator converts this set or sequence to one vector `m̃_v`. The paper tests mean pooling, self-attention, parent-conditioned cross-attention, graph-attention-style weighting, and parent-token self-attention.

For a parent with local text, the backbone receives:

`Ẽ_v = [e_write ; m̃_v ; E_v ; e_read]`.

The final read state becomes the parent memory. If the parent has no local text, the method bypasses the backbone and uses `m_v = m̃_v` directly. This shortcut changes the transformation applied at empty versus nonempty internal nodes. It is computationally sensible, but it means tree formatting can alter the representation even if descendant text is unchanged.

The internal compression ratio is extreme: an arbitrarily large subtree is summarized to one width-`d` vector. The hierarchy offers multiple retrieval levels, so the generator can receive a parent and some descendants, but top-down routing still depends on high-level vectors preserving enough query signal to keep the correct path alive.

### 2.4 Query encoding and top-down routing

The query is embedded using the same write/read pattern, but only the first half of its tokens are used: `n_q=floor(T/2)`. The paper gives no ablation or motivation for discarding the second half. This is a notable design choice because many questions place the decisive constraint or entity near the end.

At each visited parent, every child receives a scaled dot-product score:

`s(u|q) = (W_q q)^T(W_k m_u) / sqrt(d_h)`.

The projections are learned. Children are ranked locally and the top `k` from each parent advance. Starting with the root, the next candidate set is the union of selections from all current parents. Per-parent selection preserves breadth across already surviving branches, but the frontier can grow approximately as `k^depth` until leaves or a memory budget stop traversal.

The selected nodes across depths are stacked into `M_ret` and prepended to the question embeddings for answer generation. The paper does not fully specify ordering, duplicate/ancestor handling, positional encoding, or whether all frontier and ancestor memories remain in `R`. Those details can change generation behavior and the number of memory tokens.

## 3. Training objectives and optimization audit

Corpus training uses memory-conditioned language modeling. Given a node memory, the model predicts the node’s text with average token cross-entropy. An optional autoencoding loss decodes node text from the memory under a reconstruction prompt. QA fine-tuning combines answer-generation loss, routing cross-entropy, a gold-set mass objective for top-*k* selection, and optional reconstruction.

The routing objective assumes gold evidence can be mapped to leaves and propagated to ancestors. For a single gold child, cross-entropy over siblings trains the score. For multiple gold children, the selection loss maximizes total softmax mass assigned to the gold set. Inference is hard top-*k*; training uses the differentiable softmax distribution and does not backpropagate through the discrete selection.

The equations are plausible, but several semantics remain open:

- maximizing summed probability over a gold set does not force coverage of every gold child; one gold child can absorb most probability;
- mapping a span to a leaf is ambiguous when evidence crosses boundaries or requires multiple documents;
- the temperature `τ` and weights `λ_r`, `λ_s`, and `λ_ae` are unspecified;
- `V_enc`, the nodes receiving reconstruction loss, is left as an example-dependent set;
- corpus pretraining data, sampling, negative routing examples, and curriculum are not described;
- the relationship between cached memories and changing LoRA parameters during training is not explained.

The implementation note resolves one question: backbone weights are frozen and the LM is updated only through LoRA, while write/read vectors, routing projections, and applicable aggregator parameters are trained. It does not resolve how many trainable parameters are added or how the multi-stage objectives are scheduled.

## 4. Complexity and systems semantics

Routing cost is the sum of scoring the children of visited parents, `O(Σ_p c_p d_h)`. Under bounded branching and depth, the manuscript writes an upper bound proportional to `c_max d_h (k^Lmax−1)/(k−1)`. This is a worst-case frontier bound, not a guarantee of sublinear work in document size. A shallow root with hundreds of thousands of children still requires scoring them all; a large *k* or deep balanced tree can approach full traversal.

Generation prefill uses `n_q+n_ret` positions and therefore has the usual full-attention cost `O(L_LM(n_q+n_ret)^2 d)`, versus `O(L_LM(n_q+N)^2 d)` for raw full-document prefill. This explains why TTFT can fall dramatically after memories are cached.

The comparison excludes offline memory construction from online TTFT, which is appropriate for reused static documents but must be disclosed in deployment claims. Table 9 reports H²MT building an index for 30 private manuals in 977.97 seconds versus RAPTOR’s 5,382.67 seconds. H²MT’s cache is slightly larger, 63.06 MB versus 56.96 MB, because it stores 2,560-dimensional Qwen3-4B vectors rather than 768-dimensional MPNet embeddings. The total-cost crossover therefore depends on query volume, corpus churn, accelerator availability during indexing, and whether RAPTOR summaries can be shared.

The paper defines peak GPU memory as the maximum over the entire inference run, including weights, activations, KV cache, and temporary buffers. This is better than reporting only cache size. However, hardware, precision, allocator behavior, batch size, generated length, and warm-up policy are absent, so the absolute GB values cannot yet be reproduced.

## 5. Experimental design reconstructed

The main public benchmarks are NarrativeQA, HotpotQA, and QASPER from LongBench plus ORD-QA grounded in OpenROAD documentation. A fifth setting, LVS, consists of 76 private VLSI layout-verification manuals averaging 506,000 characters. The authors generate LVS question–answer pairs with DeepSeek-R1 from a node and its immediate table-of-contents children. Files are labeled “rich” or “light” using HTML structure, and question categories depend on that label.

The private dataset is relevant to the target use case but weakens independent verification. The generation prompt, item counts, manual versions, licensing, human quality control, leakage controls, and held-out split are not fully disclosed. The paper says an internal generation pipeline derives the questions and describes structural rules, but it does not provide enough material to recreate the benchmark.

Table 5 gives hierarchy statistics for four datasets: 447 OpenROAD nodes, 336,174 HotpotQA nodes, 809,278 NarrativeQA nodes, and 9,146 QASPER nodes. NarrativeQA is 98.9% leaves, while OpenROAD contains 39.8% internal nodes. The “Avg. Len.” column ranges from 3,619 to 18,409 but does not clearly state units or aggregation. This ambiguity matters because compression difficulty is a function of leaf length and number of descendants.

Baselines include the unmodified backbone, HMT, AutoCompressors, LongLLMLingua, ICAE, MemoryLLM, UltraGist/Activation Beacon, and SBERT+RAPTOR. The main ROUGE/TTFT/memory tables compare H²MT primarily with HMT and the backbone; the Llama-2-7B F1 table imports a broader set. Fairness cannot be fully audited because checkpoint versions, context budgets, training data, compression ratios, retrieved-token budgets, and generation settings are not given per baseline.

ROUGE-L is the main metric for all four principal datasets, and F1 is reported for three under Llama-2-7B. Lexical overlap is a limited proxy for answer correctness, especially when comparing latent memory with raw-text retrieval. The authors explicitly recognize that RAPTOR preserves surface forms and therefore has an advantage on overlap metrics. No semantic judge, exact match, citation recall, evidence recall, faithfulness metric, or human evaluation complements ROUGE-L.

## 6. Results, ablations, and table audit

### 6.1 Quality

Across the four main datasets, H²MT’s average ROUGE-L exceeds HMT for all five backbones shown: 35.86 versus 33.32 for Qwen2.5-14B; 30.74 versus 21.27 for Qwen2.5-7B; 34.90 versus 18.88 for Qwen3-4B; 31.43 versus 26.99 for Llama3.1-8B; and 26.02 versus 21.74 for SmolLM-135M. The gain is not universal per task. HMT is higher on HotpotQA for Qwen2.5-14B and Qwen2.5-7B, on OpenROAD for Qwen3-4B and Llama3.1-8B, and on HotpotQA for Llama3.1-8B and SmolLM.

The most striking cell is Qwen3-4B on NarrativeQA: H²MT reports 45.00 ROUGE-L versus 10.01 for HMT and 12.15 for the backbone. Such large gains deserve raw predictions and repeated-run statistics. The F1 comparison on Llama-2-7B is mixed: H²MT reaches 24.90 on NarrativeQA and 36.67 on QASPER, but 23.80 on HotpotQA. UltraGist is stronger on QASPER and HotpotQA at 37.1 and 43.3; MemoryLLM-16k also exceeds H²MT on HotpotQA at 34.03.

For the GMM-induced HotpotQA tree, H²MT reaches 46.64 ROUGE-L, 688.94 ms/document tree build, 219.72 ms TTFT, 1,661.15 ms total generation, and 15.79 GB peak memory. SBERT+RAPTOR reaches 56.39 ROUGE-L, 26,777.76 ms build, 82.71 ms TTFT, 11,864.20 ms total generation, and 16.08 GB. H²MT’s TTFT is actually higher in this row; its total generation time and build cost are much lower. Any summary saying H²MT dominates RAPTOR on TTFT would therefore be incorrect.

### 6.2 Latency and memory

Mean TTFT falls sharply against HMT for Qwen2.5-14B (2,897.49 to 468.92 ms), Qwen2.5-7B (6,398.96 to 276.89), Qwen3-4B (5,401.29 to 302.31), Llama3.1-8B (2,154.94 to 369.77), and Mistral-7B (8,111.14 to 398.85). SmolLM is the exception on NarrativeQA, where H²MT is slower, and its four-dataset mean falls only from 577.50 to 477.41.

Peak memory generally falls, but not monotonically. Qwen3-4B averages 14.60 GB versus HMT’s 14.91, with H²MT higher on HotpotQA and NarrativeQA. Mistral-7B H²MT uses 30.37 GB on QASPER versus HMT’s 27.73. The paper appropriately notes that TTFT and peak memory measure different bottlenecks.

Figures 3a and 3b sweep routing *k* on Qwen2.5-14B NarrativeQA. TTFT rises monotonically as more nodes enter prefill, while peak memory remains approximately flat. Figure 4 sweeps *k* and aggregation on OpenROAD: quality improves to a middle regime and then declines as distractors accumulate. Exact plotted coordinates are not tabulated, which limits machine comparison.

### 6.3 Aggregation and private-manual results

On OpenROAD, graph-attention pooling is best for Qwen2.5-14B (32.6) and SmolLM-135M (24.4), while cross-attention is narrowly best for Llama3.1-8B (29.0). Graph-attention also produces the lowest reported OpenROAD corpus perplexity for Qwen2.5-14B and Llama3.1-8B, 2.350 and 2.308. There is no universal winner, despite the narrative around cross-attention stabilizing the *k* sweep.

On LVS, H²MT reports test perplexity 2.62 versus HMT’s 11.93 and ROUGE-L 60.1 versus HMT’s 54.5 and the backbone’s 39.0. The very large perplexity reduction on private in-domain manuals is difficult to interpret without tokenizer, split, evaluated-token count, checkpoint-selection rules, and leakage controls. The source says the same tokenizer and evaluation script are used, which is necessary but insufficient.

### 6.4 Cross-document consistency issues

The manuscript contains several ambiguities rather than a single fatal numerical contradiction. The abstract says two structured technical-document settings, but the principal hierarchy-statistics table omits LVS. The method defines query encoding from half the question without experimental justification. The aggregation section lists five variants, while the ablation table reports four. The implementation note begins with a malformed `paragraph{...}` TeX command, suggesting the source was not fully compiled or proofread in that section. None invalidates the mechanism, but together they reinforce the need for an artifact release and a revision pass.

## 8. External context

## Source and evidence notes

### External primary-source vetting and provenance

The authors report the architecture and numerical results reconstructed above; those results were not independently rerun. The canonical arXiv record confirms the title, four authors, v1 date, Computation and Language classification, PDF, and full-paper HTML.[^1] It links no official repository, model checkpoint, dataset release, or project page. Targeted searches of the canonical record and author-linked public results found no official H²MT implementation.[^5]

OpenReview records a submission with the same authors and core title as an ICLR 2026 withdrawn submission, last modified in September 2025.[^2] That public record describes QASPER, BookSum, and technical-manual experiments and says no anonymous code URL was supplied. The May 2026 arXiv version reports a different and broader evaluation. This establishes development history, not peer-review acceptance.

The predecessor HMT is a NAACL 2025 paper with an official GitHub implementation.[^3] H²MT inherits the learned memory readout idea and contrasts its document-tree routing with HMT’s sequential memory hierarchy. RAPTOR, the closest text-retrieval comparator, has a primary publication and official implementation describing recursive clustering, summary generation, and multi-level retrieval.[^4] Those sources support the novelty boundary: H²MT’s specific contribution is learned one-vector-per-node composition plus top-down latent routing over document structure, not hierarchy, memory tokens, or recursive retrieval individually.

No same-paper DEP-A, DEP-E, Black-Lake-Data entry, or directly corresponding implementation record was verified in the repository checks. HMT and RAPTOR are predecessors/benchmarks rather than same-paper artifacts.

No DOI beyond arXiv’s repository identifier, funding statement, conflict-of-interest statement, data-availability statement, or supplementary-material declaration appears in the manuscript. Absence should not be interpreted as a disclosure of “none”; these fields are simply not reported. The complete source documents used for this review were verified locally and remain withheld from the public repository.[^6]

## 9. Code, data, and reproducibility status

Reproduction is presently blocked by missing public artifacts and configuration detail. A credible release should contain:

1. hierarchy builders for every dataset, including GMM stopping rules and table-of-contents parsing;
2. dataset versions, splits, item counts, prompts, and permissible substitutes for the private LVS corpus;
3. exact backbone and tokenizer revisions;
4. LoRA rank, alpha, dropout, target modules, optimizer, schedule, batch size, accumulation, precision, epochs, and seeds;
5. objective weights, temperature, reconstruction settings, memory budget, top-*k*, depth caps, and aggregation configuration;
6. baseline checkpoints and matched input/memory budgets;
7. generation settings, hardware, warm-up, synchronization, timing repetitions, and memory instrumentation;
8. raw predictions, evidence paths, routing traces, and per-example metrics;
9. code to reproduce every table and figure; and
10. licenses for checkpoints, generated QA, and cached latent corpora.

Without these items, the review can audit internal logic and printed results but cannot verify the claimed frontier.

## 10. Independent re-conceptualization

H²MT can be re-conceptualized as a learned, amortized multiresolution index. Each node is an index entry whose vector summarizes a region; each edge is a routing constraint; generation is a terminal consumer of a budgeted set of entries. This framing suggests three improvements.

First, separate routing memories from generation memories. A compact vector optimized to decide whether a branch is relevant need not be the best vector for reconstructing evidence. Store a small routing key at every node and one or more value vectors at leaves or high-risk nodes. This resembles key–value indexing and could reduce the burden on a single state.

Second, make pruning uncertainty-aware. Hard top-*k* ignores whether sibling scores are nearly tied. Use calibrated cumulative probability, conformal coverage, or an abstention threshold so ambiguous parents expand more children. A fixed budget can then be allocated adaptively across depth.

Third, retain public, auditable evidence handles. Latent-only generation is difficult to cite or inspect. A hybrid system can route in latent space but preserve source spans for selected leaves, retrieving raw text only when the question demands exact strings, safety-critical instructions, or traceable evidence. The system would occupy a continuum between H²MT and RAPTOR rather than choosing latent or text retrieval globally.

This re-conceptualization also changes the evaluation target. End-answer ROUGE should be decomposed into routing recall, evidence retention, answer accuracy, latency, cache cost, and update cost. A hierarchy can fail before generation, during compression, or during decoding; a single overlap score cannot localize the defect.

## 11. Research notes and critical considerations

### 11.1 Per-level routing accuracy is the missing diagnostic

Measure whether every gold evidence leaf remains reachable after each depth. Report survival probability, branching factor, calibration, and failure depth. This would test the central claimed advantage directly.

### 11.2 One-vector bottlenecks should be stress-tested with rare strings

Benchmarks should include serial numbers, exact commands, mathematical symbols, negation, cross-references, and two nearly identical sibling sections. These are the cases most likely to vanish during bottom-up aggregation.

### 11.3 Query truncation may create an avoidable blind spot

Using only the first half of a question is unsafe without an ablation. Compare first half, last half, full query, learned pooling, and dedicated embedding models, especially on questions whose final clause changes the requested scope.

### 11.4 Static-cache claims need update semantics

Document revisions should trigger recomputation only along affected ancestor paths. Cache records need document hashes, hierarchy versions, model revision, aggregation revision, and invalidation rules. Otherwise stale vectors can silently answer from old content.

### 11.5 Private-domain strength can conceal leakage

Generated QA from proprietary manuals is useful, but the generator, training corpus, and evaluation model may share knowledge. Exact source-grounding tests and evidence citations are needed to distinguish memorized domain facts from use of cached memories.

## 12. Limitations, implications, and deployment/governance applicability

The authors identify hierarchy dependence, heuristic induction for weakly structured corpora, and dilution of fine-grained evidence. Additional limitations are the missing code and configurations, no uncertainty estimates, a private benchmark, lexical-only answer metrics, no evidence-recall evaluation, no cache-update study, and no security analysis.

Deployment is most plausible for stable, heavily queried, well-structured corpora: product manuals, scientific archives, policies, and code documentation. It is less attractive for fast-changing streams, short documents, or workloads with one query per document because offline construction cannot be amortized.

Governance requires treating memory vectors as derived sensitive data. They may encode proprietary text even if inversion is imperfect. Access control, deletion, retention, model-version tracking, and membership/inversion testing should cover cached vectors as well as source documents. Latent retrieval also weakens auditability: a user cannot inspect what the model “read.” Production systems should log node identifiers and offer source-span fallback without exposing private corpus content beyond authorization.

Routing policy can create systematic blind spots. If structure or headings correlate with authoring style, language, or accessibility conventions, relevant content may be pruned unevenly. Evaluation should stratify by document structure, language, depth, and evidence position.

## 13. Falsifiable hypotheses

### Hypothesis 1: adaptive expansion beats fixed top-*k*

At equal average retrieved-memory count, a calibrated cumulative-mass router will improve gold-leaf survival by at least five percentage points over fixed per-parent top-*k*. Falsify if the difference is statistically indistinguishable on three public datasets.

### Hypothesis 2: dual routing/value memories recover lexical evidence

Separate low-dimensional routing keys and multi-vector leaf values will improve exact-string and citation questions by at least 10% relative without increasing median TTFT more than 20%. Falsify if latent capacity is not the main error source.

### Hypothesis 3: hybrid source fallback dominates either endpoint

An uncertainty-triggered raw-span fallback will match RAPTOR’s answer quality while retaining at least half of H²MT’s total-generation-time advantage. Falsify if fallback fires so often that latency converges to text RAG.

### Hypothesis 4: per-level supervision predicts end quality

Gold-path survival at the first two routing levels will explain more variance in answer correctness than final router-score margin. Falsify if generation or leaf compression dominates errors.

### Hypothesis 5: full-query encoding materially improves adversarially suffixed questions

Using the entire query rather than its first half will reduce errors by at least 20% on questions whose final clause reverses polarity or changes scope, with negligible routing cost. Falsify if the query encoder already infers the required relation from the prefix.

## 14. Replication and falsification agenda

1. Release exact code, model revisions, configurations, hierarchy files, and raw predictions for all public benchmarks.
2. Repeat every main result with at least three seeds and report confidence intervals or paired bootstrap intervals.
3. Match all baselines on backbone, retrieved-context budget, generation length, precision, and hardware.
4. Publish per-level routing survival, evidence recall, answer correctness, TTFT, total time, peak memory, cache size, and offline build cost.
5. Ablate query truncation, aggregation type, tree depth, branching, top-*k*, reconstruction loss, routing losses, and LoRA capacity.
6. Test naturally structured, flat, noisy, adversarial, and automatically induced hierarchies separately.
7. Add exact-string, negation, multi-hop, citation, and rare-evidence suites.
8. Evaluate cache invalidation and incremental updates under document revisions.
9. Compare latent-only, text-only, and uncertainty-triggered hybrid retrieval on the same trees.
10. Release a public surrogate for LVS with the identical generation and quality-control pipeline.

## 15. Durable restatement

H²MT turns a long document into a learned multiresolution cache. Offline, it builds a semantic tree and compresses every node’s local text and descendants into one vector. Online, it embeds a query, walks the tree from coarse to fine, keeps the top children under each surviving parent, and generates from the selected vectors. This is a credible way to trade exact textual evidence for lower repeated-query latency.

The paper’s reported results suggest that hierarchy-aware latent routing can substantially reduce TTFT against sequential memory baselines while improving average lexical answer quality. They also show the limit: raw-text hierarchical retrieval remains more accurate on HotpotQA, weak structure hurts, and a single vector can lose detail.

The idea is worth pursuing. The numerical claims are not yet independently actionable because the paper omits code, configurations, timing conditions, seeds, and public access to one important dataset. Until those artifacts exist, H²MT should be cited as a promising expanded preprint and an architectural hypothesis, not as a settled efficiency frontier.
