# TACO: Evolving Observation Filters for Terminal Agents

## A detailed review, technical reconstruction, and independent re-conceptualization of “A Self-Evolving Framework for Efficient Terminal Agents via Observational Context Compression”

## Executive assessment

TACO addresses a practical failure mode in long-horizon terminal agents: command output accumulates faster than the agent can use it, while simple truncation or generic summarization can erase the one error, symbol, flag, or test result that matters. The paper proposes a training-free adapter that turns compression into a library of executable, task-conditioned rules. A task begins with rules selected or generated from a global pool; uncovered high-volume observations trigger new rules; and an agent’s request for fuller output triggers conservative replacement. Successful rules gain confidence and reuse, so compression policy evolves from observable interaction traces without benchmark answers or reward labels.[^paper]

The most persuasive evidence is not a single headline score but a consistent pattern across six terminal and software-engineering benchmarks: modest accuracy gains coexist with substantial token reduction. Reported gains range from 0.36 to 6.02 percentage points depending on model and setting, while several downstream suites reduce total tokens by 12–27%. Static high-quality rules reduce tokens but do not match the full adaptive system, and removing either global-pool evolution or intra-task evolution weakens the accuracy–cost frontier. Five-run means and standard deviations improve the evidentiary quality, though the evaluation remains an artifact study rather than an independent reproduction.

The central risk is semantic over-compression. TACO’s complaint channel is useful but endogenous: an agent cannot request information it no longer realizes was present. A 200-event judge audit flags eight potential critical-loss events; manual review finds one behaviorally consequential case, and the agent’s complaint repairs the rule. This is reassuring but not a bound on rare failures. The official repository exposes code, prompts, evaluation logic, and examples under Apache-2.0, improving inspectability, but this review did not execute it.[^code]

**Overall judgment:** a strong, technically coherent systems paper with unusually explicit cost accounting and leakage boundaries. It supports the claim that adaptive observation filtering can improve terminal-agent efficiency and sometimes task accuracy. It does not establish that reward-free evolution is safe under adversarial output, that learned rules transfer across organizations, or that complaints capture all lost-critical-information events. Methodology quality: **19/20**—full sources, strong coverage, fidelity, and provenance; one point withheld because results and repository behavior were not independently reproduced.

## 1. The problem the paper is actually solving

A terminal agent repeatedly chooses a command, receives an observation, and appends both to a growing history. Much of the observation is mechanically redundant: package installation progress, compiler invocations, file listings, repeated test successes, or disassembly lines. Yet terminal traces are sparse-signal environments. An error near the end of a build, a coverage flag in one compiler command, or one symbolic call in thousands of assembly instructions can determine the next action. Context management is therefore not ordinary summarization. It is a selective preservation problem under asymmetric error: removing noise is helpful, but removing a critical token may invalidate the trajectory.

The paper’s actual intervention is at the observation layer, not the reasoning model. It does not fine-tune the backbone, alter the agent scaffold’s planning algorithm, or claim new intrinsic software-engineering ability. It inserts a rule adapter between terminal output and model context. This distinction matters when interpreting accuracy gains: better results indicate improved presentation of existing evidence, not a stronger base model.

Three requirements shape the design. First, rules must be specific enough to recognize terminal structure rather than blindly clip by position. Second, the system must adapt online because future command types cannot be exhaustively enumerated. Third, adaptation must not read hidden tests, benchmark correctness, final answers, or reward. The paper calls this reward-free unsupervised evolution: it learns only from task descriptions, commands, raw observations, compressed observations, reuse, and behavior such as a request for full output.

## 2. Formal and technical reconstruction

At step (t), let (O_t) be the raw terminal observation, (C_t) the current task and interaction context, and (R_t) the active task-level rule set. The adapter emits

\[
\widetilde O_t =
\begin{cases}
O_t, & \text{if } O_t \text{ is critical},\\
F_{R_t}(O_t\mid C_t), & \text{otherwise}.
\end{cases}
\]

In plain language, observations classified as critical bypass compression; other observations are transformed by matching rules. The equation clarifies the safety intent but delegates the hardest questions—what counts as critical and whether a rule preserved it—to implemented classifiers and rule semantics. The conditional bar means the transformation can use context; it is not merely a regular expression applied without task knowledge.

Each global rule (r) has confidence (c_r^g) and use count (n_r). Its retrieval score is

\[
R_{gs}(r)=c_r^g(n_r+1).
\]

The (+1) allows an unused rule to retain nonzero priority. Multiplication rewards both demonstrated reliability and accumulated use, but it can create a popularity loop: widely reused rules are more likely to be retrieved and thus reused again. The task initializer retrieves the top (k=30) candidates, and an LLM selects, modifies, or supplements them. Appendix sensitivity shows that raising (k) initially helps but eventually saturates while selection-token cost keeps growing.

Global convergence is tracked through top-(K) retention:

\[
\mathrm{Retention}^{(i)}_K=
\frac{|\mathrm{TopK}(R_g^{(i-1)})\cap\mathrm{TopK}(R_g^{(i)})|}{K}\times100\%.
\]

This is a stability statistic, not proof that the retained rules are correct. High retention means successive rounds select similar leading rules. It could indicate convergence, lock-in, or both. The paper pairs the metric with stable performance, which supports the convergence interpretation but does not eliminate path dependence.

## 3. Architecture and information flow

The architecture has four stateful components. The base agent produces commands and consumes compressed observations. The TACO adapter matches and executes active rules. The task-level pool holds rules selected for the current trajectory and can accept reactive additions. The global pool persists reusable rules and tracks confidence and use across tasks.

Information moves through two timescales. At task start, an initializer conditions on the task description and retrieved global rules. With history, it identifies reusable rules, modifies them if needed, and proposes task-specific additions. Without history, it cold-starts likely rules from expected command categories. During execution, matched rules compress observations. An uncovered high-output command triggers a generation prompt. Evidence that compression was too aggressive—such as re-running a command or explicitly requesting full output—freezes the problematic rule and generates a more conservative successor.

After a batch, task-level outcomes update the global pool using permitted interaction signals. The paper is explicit that benchmark reward, correctness labels, reference answers, and hidden tests are unavailable to evolution. Parallel evolution uses batch size (N=4): smaller batches propagate new rules sooner; larger batches improve throughput but delay reuse. The choice therefore encodes a latency–adaptation trade rather than a mathematically optimal constant.

## 4. The complete operational pipeline

1. Receive a task and terminal state.
2. Retrieve the 30 highest-scoring global rules relevant to the task category.
3. Ask the initializer to reuse, refine, or create structured task-level rules; cold-start when the global pool is insufficient.
4. Let the base agent issue a terminal command and collect the full raw observation.
5. Bypass compression for a critical observation; otherwise match a rule and apply its semantic filter.
6. Deliver only the compressed observation to the reasoning context while retaining the raw trace for safety and evaluation.
7. If a long output is uncovered, generate a reusable rule for that command family.
8. If subsequent behavior suggests missing information, freeze the aggressive rule and create a conservative replacement.
9. Continue until task completion, accounting for both backbone tokens and auxiliary rule-evolution calls.
10. Merge eligible task rules into the global pool, update confidence and counts, and repeat in four-task batches.

This reconstruction exposes two operational invariants. Raw output must remain recoverable, otherwise complaints cannot restore it. Rule execution must also be sandboxed because a generated filter is code-like policy operating on potentially hostile terminal text.

## 5. Re-conceptualization

A useful independent framing is **an online compiler for observation contracts**. Each terminal command establishes an implicit contract: preserve exit status, errors, changed files, task-specific markers, and uncertainty; summarize or discard repetitive details. TACO compiles those contracts into reusable filters and updates them when downstream behavior reveals a mismatch.

This framing improves conceptual precision. “Compression” sounds like reducing length, but the system’s objective is preserving decision sufficiency. A good filter may keep a long error block and remove a short but redundant listing. The global pool resembles a policy cache, the task pool resembles a specialized executable plan, and complaints resemble counterexamples in counterexample-guided refinement. Unlike standard CEGAR, however, counterexamples are detected by the same fallible agent whose input was filtered.

The framing suggests a safer successor: every rule should declare preservation invariants, a command-domain type, tests generated from raw traces, and a reversible pointer to omitted spans. Selection would optimize expected context value subject to invariant checks. That moves the system from free-form semantic filtering toward typed, auditable observation transformations.

## 6. Experimental design reconstructed

The study covers TerminalBench 1.0 and 2.0, SWE-Bench Lite, CompileBench, DevEval, and CRUST-Bench. These suites span interactive terminal tasks, repository repair, compilation, code generation, and Rust debugging. Multiple backbones are paired with terminal-agent scaffolds, including Qwen3 variants, Qwen3-Coder-480B, DeepSeek-V3.2, and MiniMax-M2.5. Local inference uses a 20× NVIDIA H800 cluster. Results are averaged over five independent runs unless otherwise stated.

The primary endpoints are task accuracy and token consumption. Token totals include agent execution plus auxiliary LLM calls for rule selection, generation, and revision; the appendix reports auxiliary overhead below 2%. This inclusive denominator is a major strength because an adaptive compressor could otherwise hide its own cost. Fixed-token-budget curves test whether accuracy gains persist under matched spending.

Comparators include the uncompressed baseline, manually reviewed high-quality static rules, LLM-generated static rules, and LLM summarization. The high-quality set contains 200 structured rules derived from public trajectories with LLM assistance and human review. Three computer-science master’s students were compensated at about $19 per hour. The adaptive system is further tested with global-pool evolution removed and intra-task evolution removed.

The paper audits rule quality through 200 uniformly sampled compression events judged on critical-information preservation, noise removal, and useful-context retention. It manually inspects all eight events flagged for potential critical loss. Case studies trace package installation, instrumented compilation, and binary disassembly. Best-of-(K) pass rates, rule reuse, pool retention, parameter sensitivity, and closed-source model results provide secondary evidence.

What is missing includes independent execution, confidence intervals for every derived comparison, adversarial terminal-output tests, cross-project privacy isolation, and prospective evaluation on tasks gathered after the global pool was frozen.

## 7. Results: what is reported and what it means

Across the six headline benchmarks, TACO’s MiniMax-M2.5 results move from 56.30 to 57.12 on SWE-Bench Lite, remain 75.00 on CompileBench, rise 38.10 to 39.74 on DevEval, 47.00 to 48.05 on CRUST-Bench, 42.30 to 45.25 on TB 1.0, and 42.80 to 44.16 on TB 2.0. Corresponding total-token changes are 307.61M to 270.53M, 14.55M to 11.41M, 36.72M to 26.82M, 163.53M to 134.97M, 29.74M to 23.43M, and 113.74M to 110.63M. Thus the strongest efficiency result is DevEval’s 27.0% reduction; the weakest is TB 2.0’s 2.7%.

Across TB 2.0 backbones, accuracy gains vary rather than scale monotonically. Qwen3-Coder-480B rises 23.3 to 25.8, DeepSeek-V3.2 40.6 to 42.7, MiniMax-M2.5 42.8 to 44.1, Qwen3-32B 3.9 to 7.4, Qwen3-14B 4.0 to 6.1, and Qwen3-8B 1.4 to 3.6. Token-per-step sometimes increases for smaller models even when accuracy rises, warning against the blanket claim that TACO always lowers every cost measure.

Five-run tables generally show lower dispersion with TACO. On TB 2.0, MiniMax changes from (42.80\pm2.54) to (44.16\pm0.86); Qwen3-Coder-480B from (23.90\pm1.48) to (25.86\pm0.78). These are run-level standard deviations, not standard errors, and task-level dependence is not modeled.

Case studies make the mechanism plausible. A package-install continuation shrinks from 10,071 to 73 characters after an evolved lifecycle-log rule. A build filter removes copied-file noise while retaining the final command’s coverage flags. A disassembly rule is reused 18 times and preserves labels and calls while removing repetitive instruction detail. These are explanatory examples, not randomly selected causal estimates.

## 8. Ablations and causal evidence

On TB 2.0 with DeepSeek-V3.2, the baseline is 40.6% accuracy and 99.60M tokens. Removing global rule-pool evolution gives 40.4% and 81.57M; removing intra-task evolution gives 38.9% and 69.02M; the full system gives 42.7% and 81.45M. This pattern supports complementarity: global reuse helps avoid relearning, while reactive evolution handles novel outputs. It also reveals that the most aggressive token reduction is not the best task policy.

Static comparisons reinforce the point. Baseline accuracy is (23.9\pm2.9). High-quality static rules reach (24.3\pm2.2) with 17.9% token reduction; LLM-generated rules fall to (19.71\pm1.6); LLM summarization reaches (20.3\pm3.2) with 21.3% reduction; TACO reaches (25.9\pm1.5) with 10.78% reduction. Adaptive preservation, not maximal shortening, appears responsible for the better frontier.

Online rules can be reused without losing performance: reported reuse variants approximately match or exceed online evolution for DeepSeek-V3.2, Qwen3-Coder-480B, and MiniMax-M2.5. Parameter analysis justifies (k=30) and (N=4) empirically, but only on one benchmark/backbone slice. The eight flagged loss events provide a safety ablation of sorts; one complaint-driven repair demonstrates the channel works at least once, not that it has complete recall.

## 10. External views and field context

The official arXiv record identifies version 3 as the reviewed stable source.[^paper] The official repository exposes a broad implementation under Apache-2.0, with code, prompts, examples, evaluation assets, and documentation.[^code] That is stronger than a paper-only artifact, but repository presence is not reproduction.

TerminalBench provides the directly relevant interactive terminal-evaluation context.[^terminalbench] SWE-Bench Lite supplies a widely used repository-repair slice, though benchmark contamination and infrastructure variability remain general concerns.[^swebench] These primary sources support the existence and intended scope of the evaluations; they do not independently validate TACO’s numbers.

The closest intellectual relatives are context-management and test-time adaptation systems, but TACO’s distinguishing choice is executable observation-specific rules plus reward-free cross-task memory. ## 11. Research notes and critical considerations

**Source integrity.** The reviewed PDF is 27 pages and matches full-paper HTML for arXiv:2604.19572v3. Introduction, formal method, six-benchmark evaluation, appendices A–N, 15 tables/listings, six figures, prompts, ethics, compute, and statistical details were available.

**Novelty.** Individual ingredients—rule filtering, retrieval, confidence scoring, and complaint-based recovery—are familiar. The novelty lies in their terminal-specific assembly, persistent reward-free pool, and unusually complete evaluation of efficiency and failure events.

**Metric semantics.** “Accuracy” inherits each benchmark’s pass criterion and is not a calibrated probability. Token reduction is end-to-end and therefore meaningful, but it is not directly equivalent to latency, energy, or monetary cost across heterogeneous models and serving stacks.

**Reproducibility.** The paper reports (k=30), batch (N=4), five runs, model/scaffold pairings, local H800 serving, prompts, and cost accounting. Missing or fragile elements include exact immutable model snapshots, every container dependency, a frozen repository release, and an independent trace replay.

**Safety.** Terminal output is an untrusted input channel. It can contain prompt injection, ANSI/control sequences, fabricated success markers, or deliberately shaped text that causes a filter to discard evidence. The rule-generation LLM and rule executor require strict separation from command authority.

**Disclosure and broader impact.** The paper reports no private user data, describes annotator compensation, acknowledges that risks derive largely from underlying agents, and recommends sandboxing plus human oversight. Funding and conflicts should be read from the canonical paper; this review found no basis to infer undeclared relationships.

## 12. Potential implications

For agent engineering, TACO suggests context management should be treated as a stateful subsystem with its own evaluation, not a tokenizer-side truncation option. A modest accuracy gain paired with lower total tokens can materially improve throughput when terminal workloads are long.

For research, the paper creates a useful unit of analysis: the compression event. Event-level raw/filtered pairs enable tests for preservation, usefulness, adversarial robustness, and delayed consequences. That supports more rigorous evaluation than judging only final task success.

For governance, persistent global rules become learned operational policy. They need provenance, tenancy boundaries, versioning, rollback, and retention limits. A rule learned from one customer’s commands must not expose command literals or assumptions to another. The global pool should be regarded as potentially sensitive derived data even if raw traces are not stored in the public repository.

## 13. New hypotheses derived from the paper

1. **Invariant hypothesis:** typed preservation contracts will reduce critical-loss rate without materially eroding token savings compared with free-form rules.
2. **Counterexample hypothesis:** complaints plus automatic raw/filtered differential tests will find more unsafe rules than complaints alone.
3. **Transfer hypothesis:** command-family rules transfer well within a toolchain but degrade sharply across operating systems, locales, and version changes.
4. **Calibration hypothesis:** global confidence should decay with environment drift; fixed cumulative use counts will over-rank stale rules.
5. **Adversarial hypothesis:** outputs designed to imitate low-value progress lines can evade semantic filters and cause materially higher loss rates than benign benchmarks.
6. **Utility hypothesis:** decision-sufficiency metrics predict task outcome better than compression ratio or judge-rated readability.

Each is a reviewer proposal, not a paper result.

## 14. Deployment and governance considerations

Production use should preserve an immutable raw-observation log under access control, store the exact rule version applied at each step, and expose one-click restoration of omitted spans. Generated rules should run in a non-executable, bounded transformation language rather than arbitrary code. Tenant-specific and shared pools should be physically and logically separated.

Promotion should require replay tests across success, failure, warning, localization, binary-output, and adversarial fixtures. Monitoring should track compression ratio, complaint rate, raw-restoration rate, downstream re-execution, rule age, environment version, and task failure following compression. Automatic rollback should trigger on excess complaints or divergence from uncompressed shadow runs.

The adapter must never gain shell authority. Prompts and terminal data should be escaped and delimited, control sequences normalized, secrets redacted before persistence, and all generated transformations constrained to reading a supplied byte stream. Human review remains warranted for high-impact administrative, security, or production operations.

## 15. Replication and falsification agenda

1. Freeze the official repository commit, model weights, prompts, containers, and benchmark snapshots.
2. Reproduce the six headline tables with five seeds and publish per-task outcomes, total tokens, auxiliary tokens, latency, and energy.
3. Replay identical raw traces through baseline, static rules, summarization, and TACO to isolate observation-policy effects from trajectory divergence.
4. Label a stratified sample of at least 2,000 events with independent annotators blind to method; report precision and recall for critical-loss detection.
5. Inject adversarial errors, misleading progress text, ANSI sequences, non-English messages, and tool-version drift.
6. Run uncompressed shadow agents on a subset and compare next-action distributions and final success.
7. Freeze the global pool, evaluate chronologically newer tasks, then test transfer across operating systems and organizations.
8. Ablate complaint signals, raw restoration, confidence decay, use-count scoring, (k), and batch size with matched inference budgets.
9. Publish every failure trace where compression changes a correct baseline trajectory into an incorrect one.

The central falsifier is clear: if matched-budget, independently reproduced trials show that gains vanish or critical-loss frequency exceeds an operationally acceptable bound, the claimed frontier does not hold.

---

## 16. Durable restatement

TACO is best understood as a persistent, training-free observation-policy layer for terminal agents. It retrieves and specializes semantic filters, creates rules for new output patterns, repairs rules after observable complaints, and carries successful rules across tasks without consulting benchmark rewards. The complete paper reports small-to-moderate accuracy improvements and meaningful total-token savings across heterogeneous terminal benchmarks, backed by ablations, five-run statistics, case studies, and a critical-loss audit.

Its durable contribution is the argument that context compression should learn *what a tool’s output means for the next decision*, not merely how to shorten it. Its durable limitation is that the agent and judge detecting information loss may share the same blind spot. Any deployment should therefore combine adaptive filtering with reversible raw evidence, typed invariants, independent replay, provenance, and strict execution isolation.
