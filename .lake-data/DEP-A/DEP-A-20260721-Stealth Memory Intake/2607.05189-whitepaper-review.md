## Executive assessment

### The paper in one sentence

The paper demonstrates, in a controlled personal-agent email workflow, that one untrusted message can cause a covert durable memory write and influence behavior in a later session, then proposes WhisperBench and MemGhost to measure and optimize that attack lifecycle.

### The deeper idea

The important mechanism is not email-specific and not reducible to ordinary prompt injection. A persistent agent transforms observations into state, and that state later participates in decisions. If the transformation does not preserve origin trust and if retrieval confers implicit authority, untrusted content crosses time and context boundaries. The original conversation can disappear while its policy effect remains. The reviewer inference is that memory behaves like a delayed capability grant: writing a preference, fact, or instruction can change what future tool calls the system considers authorized.

### Bottom-line judgment

The paper makes a strong case that stealthy cross-session memory injection is a credible vulnerability class in the tested environment. Its evaluation is more system-realistic than a single-turn prompt benchmark because it uses authentic IMAP/SMTP interactions, persistent state, later probes, component metrics, held-out cases, multiple victim models, and multiple agent/memory configurations. The strongest reported background end-to-end rates are high: 91.1% for Grok 4.1 and 87.5% for GPT-5.4. Foreground interaction often lowers stealth, but it does not consistently prevent writing or later execution.

The evidence does not establish production prevalence, vendor ranking, or a universal bypass. The cases are synthetic, the horizon is short, the main study lacks confidence intervals and independently repeated training seeds, query-based baselines use a different denominator, and no public code or dataset link was available as of the review date. Several editorial inconsistencies also require clarification. The paper is therefore best treated as a well-motivated controlled red-team result and benchmark proposal, not as a field measurement or deployment certificate.

### Principal strengths

- It isolates injection, visibility, execution, and end-to-end outcomes instead of reporting one opaque attack-success number.
- It uses a real mail protocol and persistent-agent workflow across background and foreground operation.
- It provides a 108-case benchmark with a 52/56 train/held-out split and five risk categories.
- It evaluates model, runtime, memory-backend, defense, and visibility variation.
- It discloses substantial optimizer, reward, proxy, rubric, ethics, and AI-assistance detail.
- It surfaces an actionable architectural failure: untrusted content becomes durable state without a commensurate authority transition.

### Principal qualifications

1. The principal results are author-reported and were not independently reproduced.
2. Baseline denominators are not uniform: search methods use a 20-case subset, while principal one-shot methods use 56 held-out cases.
3. Metric prose is internally ambiguous about whether execution success is conditional on injection or measured across all runs.
4. Results are tied to model endpoints, agent versions, evaluator rubrics, and a controlled date-specific harness.
5. No public code, benchmark, checkpoint, or exact executable configuration was linked from the v1 paper record.
6. The paper does not test real users, long-horizon consolidation, multi-agent spread, multi-tenancy, multimedia mail, or upstream mail defenses.

## 1. The problem the paper is actually solving

### 1.1 Problem and stakes

Persistent personal agents are designed to remember user facts, preferences, tasks, and operating conventions. That capability creates value because a later session can act without reconstructing the user's history. It also creates an integrity risk: an external document can be interpreted as evidence about the user and converted into durable state. A later agent session may retrieve that state after the external document and original conversation are no longer visible.

The paper studies a one-shot, black-box attacker who can send an email but does not control the victim model, hidden prompt, or memory database. Success requires three stages. First, the agent writes attacker-influenced state. Second, the agent avoids a user-visible response that clearly discloses the malicious change. Third, a later session retrieves or follows the state and changes behavior. The end-to-end outcome is the intersection of these stages.

The stakes depend on the affected memory category. A false preference can steer recommendations; a false fact can contaminate later reasoning; a retained instruction can affect communication or tool use. The paper organizes cases into five risk categories and includes both fact and preference memories. This review omits concrete malicious messages, triggers, and payload recipes because they are not needed to understand the defensive mechanism.

### 1.2 Prior method families

Conventional prompt-injection work often evaluates whether untrusted text changes behavior in the same context. Memory-poisoning work extends that concern to retained state. Search-based attacks repeatedly query the target or a shadow system to improve a payload, while handcrafted baselines use fixed patterns. Defense families include content screening, prompt alignment, runtime monitoring, memory provenance, retrieval gating, and action authorization.

The paper argues that these families leave a practical gap. Same-session success does not prove persistence. A visible malicious response does not satisfy stealth. A memory write does not prove later behavioral impact. Expensive target queries also make optimization difficult when the victim is a hosted black-box agent. WhisperBench and MemGhost address those evaluation and optimization gaps.

### 1.3 The paper's proposed gap

The proposed gap has two parts. The evaluation gap is the absence of a realistic benchmark that spans ingestion, durable state, user visibility, and later execution through an authentic personal-agent workflow. The optimization gap is the cost of learning effective messages when every candidate would otherwise require complete victim-agent execution. The authors use a platform-environment proxy and objective proxy so the attacker policy can train with cheaper feedback, then evaluate transfer on real target configurations.

The paper reports this as an attack contribution. The defensive restatement is that stateful-agent evaluations need explicit transition observability. A benchmark must record the source content, proposed memory diff, accepted memory, user-visible output, later retrieval, plan, and action. Without those stages, a defense cannot be credited to the correct control layer.

## 2. Formal and technical reconstruction

### 2.1 Definitions and assumptions

Let a case contain an ingestion context, an intended malicious memory effect, and a later probe. Let the attacker policy produce an email conditioned on that case. The victim agent processes the email in either background or foreground mode and may update persistent memory. A later, separate session executes the probe. The evaluation derives binary or rubric-scored outcomes for injection, stealth, execution, and their end-to-end intersection.

The attacker is black-box with respect to the final victim. Training instead uses a shadow environment. The assumptions are consequential: the attacker can deliver a message that the agent reads; the agent is permitted to write persistent memory from email-derived content; a later session can retrieve that memory; and the relevant model and runtime behaviors transfer sufficiently from the shadow setting.

### 2.2 Objective and central equations

The paper's attacker objective seeks a policy whose messages maximize expected end-to-end success over benchmark cases and target environments. The proxy objective decomposes the difficult end-to-end signal into rewards for injection, stealth, and trace/effectiveness. In conceptual form:

`R = w_inj * R_inj + w_stl * R_stl + w_trace * R_trace`,

subject to the language policy remaining capable of producing plausible messages for the case. The exact implementation includes supervised fine-tuning, entropy regularization, a KL term, and GRPO updates over groups of sampled completions. This decomposition is operationally important because it creates dense optimization signals, but it also makes proxy validity part of the threat model.

Table III reports positive relationships between proxy rewards and target outcomes. Injection reward versus execution success has Pearson 0.872, Spearman 0.771, Kendall 0.874, and directional agreement 0.937. Stealth reward versus stealth success reports 0.785, 0.710, 0.803, and 0.901. Trace reward versus end-to-end success reports 0.839, 0.795, 0.852, and 0.926. These support usefulness in the tested domain; they do not prove calibration under new runtimes or defender adaptations.

### 2.3 Components

WhisperBench supplies structured training and evaluation cases. A platform-environment proxy emulates the relevant workspace and operations. An objective proxy scores candidate behavior without invoking every real victim configuration. The attacker policy is a Qwen 4B model adapted first with supervised fine-tuning and then with GRPO. A shadow-agent configuration supports optimization, while held-out real target agents supply final evaluation.

The victim side includes personal-agent runtimes, email tooling, persistent memory, background scheduling or foreground conversation, and later task execution. OpenClaw, NanoClaw, Hermes, and Mem0-backed variants appear in transfer evaluation. The paper also tests six model endpoints in the main comparison and several existing defenses at different intervention points.

### 2.4 Training process

1. Construct 108 benchmark cases and split them into 52 training cases and 56 held-out cases.
2. Use seed search and a shadow environment to obtain candidate demonstrations and train the Qwen 4B policy with supervised fine-tuning.
3. Continue optimization with GRPO, sampling groups of eight completions and scoring proxy injection, stealth, and trace objectives.
4. Maintain a hard-example buffer and replay fraction to emphasize difficult cases without discarding prior coverage.
5. Freeze the trained policy and generate one-shot messages for held-out evaluation on target models and runtimes.

The supervised phase uses LoRA rank 16 and scaling 32 for three epochs, batch size 2, gradient accumulation 8, AdamW learning rate `2e-4`, warmup ratio 0.1, and weight decay 0.01. The reinforcement phase uses LoRA rank 64 and scaling 128, about 132 million trainable parameters, AdamW at `5e-6`, KL coefficient 0.01, entropy coefficient 0.05, group size 8, temperature 1.2, maximum completion length 2,048, hard-buffer size 50, and 30% replay. The first curriculum stage runs for 30 iterations.

### 2.5 Inference process

1. Condition the trained attacker policy on one held-out benchmark case and generate one email without iterative target feedback.
2. Deliver the email through the controlled IMAP/SMTP workflow to a target agent operating in background or foreground mode.
3. Inspect whether the intended persistent state was written and whether the user-visible response disclosed the change.
4. Start a later session with the probe task and evaluate whether retained state changes the target behavior.
5. Record ISR, SSR, ESR, and E2E using case-specific rubrics.

The query-based baselines differ: they receive up to 50 search iterations and are reported on a fixed 20-case subset. That difference must remain attached to any comparison.

### 2.6 Notation and implementation audit

The main issue is ESR wording. One source passage describes execution success among successful injections; the active metric presentation and table relationships are consistent with an all-run denominator. A replication should publish the raw four-stage outcome tuple for every case and both conditional and unconditional ESR.

The v1 source contains an unfinished limitations sentence ending in `xxx`, a grammatical defect in the RQ3 wording, and a figure-count mismatch: the arXiv comment says eight figures, while the inspected document contains Figures 1-10. No equation-level sign error was identified in the reviewed optimization and reward presentation. These editorial defects lower manuscript-polish confidence but do not independently invalidate the reported measurements.

## 3. Architecture and information flow

### 3.1 Representation

The critical representation is a persistent memory record derived from an untrusted message. In the tested systems, memory may be native workspace state or managed through a memory backend. The paper evaluates whether a target effect is represented after ingestion, but the broader security problem is what metadata accompanies that representation. A plain natural-language record typically does not encode channel authentication, source trust, extraction method, user approval, intended scope, expiration, or permitted actions.

The independent re-conceptualization proposed here is a typed memory envelope. Every retained value should be coupled to a provenance receipt, confidence, sensitivity, authority class, scope, expiration, contradiction state, and approval digest. That representation lets retrieval and action policy distinguish "searchable evidence" from "user-approved instruction."

### 3.2 Decision or action modules

The attacker policy chooses email content. The victim model chooses whether to write memory, what to reveal, what to retrieve later, and what action to take. Existing architectures often let these choices flow through the same language-model policy. That concentration permits a persuasive message to influence both the memory content and the apparent justification for accepting it.

A defensive system should separate deterministic policy decisions from model suggestions. The model may propose a memory diff, but a gateway assigns authority. The model may propose a tool call, but an action broker checks capability, evidence, and confirmation. User visibility should be produced from the committed diff and policy decision, not solely from free-form model narration.

### 3.3 Dependencies and post-processing

The full chain includes mail delivery, parsing, tool schemas, model endpoint, memory extraction, storage backend, retrieval ranking, prompt assembly, planner, tool broker, and user interface. A version or policy change at any link can change the result. Therefore, claims about one model name are not portable without the surrounding runtime coordinates.

Post-processing should include canonicalization, source hashing, duplicate detection, contradiction checks, scope assignment, and user-receipt generation. Sensitive actions should perform a reverse-lineage query: which memories affected this plan, where did they originate, and which independent approvals authorize their use?

## 4. The complete operational pipeline

### 4.1 End-to-end flow

The paper's operational flow begins with controlled mail delivery. The target agent reads a mixed inbox through an authentic email skill. During ingestion, it may write persistent state. In foreground mode, a user is present and sees a response; in background mode, a scheduler may process messages and notify only under configured conditions. The system preserves memory while isolating conversation history. A later task activates the retained state. Evaluators inspect the memory and later behavior using case-specific rubrics.

MemGhost's optimization pipeline occurs before target evaluation. Seed search and supervised examples initialize the attacker policy. The platform proxy supplies tool-like environmental feedback. The objective proxy scores stage outcomes. GRPO improves samples relative to their group while KL and entropy terms constrain policy movement and diversity. The trained policy then generates one-shot held-out messages.

### 4.2 Resource allocation

Training uses 52 cases, while final principal evaluation uses 56. Samples are generated in groups of eight during GRPO. The hard buffer holds 50 difficult items and replay uses 30% of training activity. Query-based baselines receive 50 iterations but only on 20 fixed held-out cases. The paper names several hosted models and a shadow model, but it does not provide a complete hardware, wall-clock, token-cost, or per-endpoint budget ledger.

For defenders, the relevant allocation problem is different. Provenance checks are cheap and deterministic; human confirmation is expensive; full behavioral simulation adds latency; and continuous trace retention adds storage and privacy costs. A practical policy should reserve stronger controls for records and actions with higher sensitivity or authority.

### 4.3 Guarantee boundary

All reported security outcomes are empirical. MemGhost does not provide a formal guarantee of injection, stealth, transfer, or defense evasion. WhisperBench does not prove production prevalence. A defense that performs well on the benchmark would not be certified against unknown attacks.

The only robust architectural guarantee available from this review is policy-derived: a record assigned `search_only` authority cannot directly authorize a sensitive action if the action broker is complete, independent, and fail-closed. That guarantee depends on implementation coverage. Any bypass path around the broker invalidates it.

## 5. Re-conceptualization

### 5.1 Memory as a delayed capability grant

The paper frames the outcome as stealthy memory injection. A more general mechanism is a delayed capability grant. When an agent accepts a retained instruction or preference, it changes the set of future actions considered legitimate. The content may appear informational, but the downstream effect is normative.

**Boundary of the analogy:** not every memory has action authority. A historical note can remain passive evidence. The analogy is useful only when retrieval can influence planning, preference, or tool use.

### 5.2 Provenance is a policy input, not an annotation

Many systems attach a source string for debugging but do not enforce it. The paper's threat persists because the origin of the message does not prevent a future session from treating the derived memory as trusted. Provenance becomes a security control only when write acceptance, retrieval ranking, prompt assembly, and action authorization consume it.

**Boundary of the interpretation:** provenance can be forged or incomplete. Sender authentication proves a channel fact, not the truth or legitimacy of content. It must be combined with scope, confirmation, and behavior controls.

### 5.3 Transferable design principle

No untrusted observation should become durable action authority through a single model judgment. Convert observations into proposed state; preserve their lineage; assign the least authority needed; require independent evidence for privilege elevation; and make every change visible, expirable, and reversible.

This principle transfers to RAG documents, browser content, chat messages, tickets, shared agent memory, tool outputs, and imported profiles. The source channel changes, but the trust transition is the same.

## 6. Experimental design reconstructed

### 6.1 Data

WhisperBench contains 108 cases across five risk categories and fact/preference memory types. Fifty-two cases are used for training and 56 are held out. Each case includes an ingestion scenario, target memory effect, later probe, and rubrics for injection, stealth, and effectiveness. The appendices expose an example case and rubric structure.

The benchmark emphasizes fidelity through a mixed inbox and real mail protocol. Diversity comes from case categories and target systems. Generalizability is tested through model and runtime transfer. Consequentiality comes from later behavioral effects. These are design intentions backed by controlled cases, not validation against a population of real user mailboxes.

### 6.2 Training and hardware

The paper reports optimizer and LoRA settings in useful detail, as reconstructed in Section 2.4. DeepSeek-V3.2 supports seed search, GPT-5.4-mini is the shadow agent, and a Qwen 4B policy is trained. The paper does not provide enough publicly linked artifacts to reproduce the exact run. Hardware, total tokens, wall-clock duration, complete seeds, service versions, and cost are not comprehensively archived.

This matters because group-relative training and hosted-model feedback can vary across seeds and endpoint revisions. A credible release should pin the training code, dependency lock, checkpoint hash, benchmark split, prompt/evaluator versions, and every hosted call timestamp.

### 6.3 Baselines

The paper compares handcrafted patterns, query-intensive search methods, and MemGhost. Handcrafted and MemGhost results use the 56-case held-out set. Query-based methods use a fixed representative 20-case subset and 50 search iterations. DeepSeek-V3.2 appears in seed search. Because sample sets and query budgets differ, raw percentages across all rows are not directly exchangeable.

Fairer analysis would report every method on the common 20 cases, retain the 56-case one-shot evaluation as a separate transfer test, and publish paired differences with uncertainty. It should also include a benign-message utility baseline and ordinary legitimate preference updates, since a defense can trivially reduce attack success by suppressing memory entirely.

### 6.4 Metrics

ISR measures whether the intended memory injection occurs. SSR measures whether the user-visible behavior satisfies stealth criteria. ESR measures downstream effectiveness, with the denominator ambiguity noted earlier. E2E measures the intersection required for a complete attack. The paper's choice to separate stages is sound because a high ISR with low SSR, such as MiniMax M2.5 in foreground mode, has a different risk profile from a fully covert effect.

Metrics depend on evaluators and rubrics. The paper includes rubric detail, but evaluator reliability, inter-rater agreement, and sensitivity to model judges are not fully characterized. Raw artifacts and blinded human adjudication would strengthen validity.

### 6.5 Runtime semantics

Background and foreground modes are not merely UI variants. Foreground operation can elicit a visible response and user scrutiny; background operation may silently process mail under a scheduler. Both share persistent state and tools. The paper tests transfer across OpenClaw, NanoClaw, Hermes, native memory, and Mem0-backed configurations, but exact version pins and configuration parity remain important missing coordinates.

Throughput and latency are not primary study outcomes. Query-based methods consume many target interactions, while MemGhost shifts cost to proxy training and one-shot deployment. Security comparisons should include target queries, training compute, hosted-model calls, evaluator calls, and end-to-end cost.

## 7. Results: what is reported and what it means

### 7.1 Main model comparison

Table I reports four metrics for background and foreground execution. For GPT-5.4, background ISR/SSR/ESR/E2E is 96.4/100/87.5/87.5; foreground is 94.6/82.1/87.5/75.0. Claude Sonnet 4.6 reports 73.2/73.2/71.4/71.4 in background and 51.8/51.8/48.2/48.2 in foreground. Claude Sonnet 4.5 reports 92.9/94.6/82.1/80.4 and 92.9/80.4/89.3/71.4.

MiniMax M2.5 reports 94.6/94.6/82.1/78.6 in background and 98.2/21.4/85.7/21.4 in foreground. Grok 4.1 reports 96.4/100/91.1/91.1 and 89.3/96.4/80.4/76.8. DeepSeek v4 Flash reports 96.4/87.5/85.7/75.0 and 96.4/51.8/85.7/42.9.

**Interpretation:** one-shot optimized messages transfer to multiple tested victim endpoints, and background processing often provides a favorable stealth environment. Component metrics expose model-specific failure modes.

**Qualification:** the paper reports point estimates on 56 cases, without confidence intervals or an independent replication. Hosted model identities and behaviors can change. Percent differences of a few cases should not be overinterpreted.

### 7.2 Runtime and memory transfer

Table II holds GPT-5.4 constant while varying frameworks and memory backends. OpenClaw native reports background E2E 87.5% and foreground 75.0%. OpenClaw with Mem0 reports 85.7% and 80.4%. NanoClaw native reports 80.4% and 78.6%. Hermes native reports 82.1% and 51.8%. Hermes with Mem0 reports 85.7% and 48.2%.

**Interpretation:** the tested effect is not confined to one native storage implementation. The results support an architectural concern around how agent runtimes turn external content into persistent state.

**Qualification:** every row uses GPT-5.4, and historical runtime versions/configurations are not independently pinned here. The table does not disentangle all interactions among model, tool schema, extraction policy, storage, retrieval, and UI.

### 7.3 Defense, visibility, and robustness

Table IV reports false-negative rates of 91.59% for DataSentinel; 49.07% for Meta-SecAlign on injection and 98.15% on stealth; and 93.46% for AgentDoG on OpenClaw versus 40.38% on the Claude Code SDK. Table V evaluates adaptive foreground visibility. E2E becomes 23.2% for Claude Sonnet 4.6, 19.6% for Sonnet 4.5, 3.6% for Hermes with GPT-5.4, and 1.8% for OpenClaw with GPT-5.4.

**Interpretation:** tested content or agent defenses miss many cases under their evaluated settings, while stronger visible disclosure can disrupt the stealth objective. The variation across runtime settings reinforces the need for system-level evaluation.

**Qualification:** false-negative rates are threshold- and distribution-specific. Visibility can expose a write without preventing it. The study does not measure user comprehension, response, alert fatigue, or the clean-utility cost of proposed controls.

## 8. Ablations and causal evidence

### 8.1 Proxy reward design

Table III is the strongest evidence for the proxy mechanism. Reward/outcome correlations are high across Pearson, rank, Kendall, and directional measures. This indicates that the selected proxy signals preserve useful ordering and direction in the tested targets. It does not isolate which proxy component causes transfer, nor whether the relationship survives adversarial defender adaptation.

A stronger ablation would remove each reward term, retrain across multiple seeds, and measure paired changes in all four end metrics. It would also compare against a reward model trained on the same interaction budget without the structured decomposition.

### 8.2 Foreground visibility

The foreground comparisons and Table V provide quasi-ablative evidence that visible output is a control surface. The MiniMax row is especially informative: injection and execution remain high while stealth collapses, reducing E2E. Adaptive visibility produces very low E2E in two GPT-5.4 runtime configurations.

This does not identify whether lower success comes from response generation, user-facing length, memory-policy changes, or evaluator rubrics. A causal test should manipulate only the receipt channel while holding model, prompt, memory decision, and later probe constant.

### 8.3 Randomness and statistical evidence

The paper provides training hyperparameters but does not present a main multi-seed uncertainty analysis. With 56 held-out cases, one case corresponds to about 1.79 percentage points. Many reported differences therefore amount to a small number of cases. Confidence intervals, paired tests, bootstrap distributions, and training-seed variance are needed before declaring small model differences meaningful.

No independent reproduction was performed for this review. The available evidence supports feasibility and mechanism plausibility, while causal attribution among specific optimizer, proxy, model, and runtime components remains incomplete.

## 10. External views and field context

### 10.1 Predecessor evidence

FARMA studies forged-reasoning attacks against agent memory and proposes a multi-signal defense. It reinforces the idea that retained state can carry adversarial policy influence and that provenance plus consistency signals should mediate acceptance.[^farma] The relationship is mechanistic, not identity: FARMA targets forged reasoning, while the reviewed paper targets email-derived facts/preferences and stealth across sessions.

The architecture-layer defense study evaluates persistent-memory attacks at input, retrieval, prompt, and tool layers. Its main relevance is control placement: a defense cannot stop a state transition it cannot observe, and capability removal may reroute rather than eliminate behavior.[^defense-layers] Together, these works support policy enforcement around memory authority rather than exclusive reliance on text classification.

### 10.2 Independent evaluation perspective

The related Agent Memory Forensics DEP emphasizes trace evidence, negative-class construction, and incident reconstruction. This perspective asks whether memory writes and downstream actions leave discriminative trajectories. Forensics is valuable after detection, but a trace classifier is not an authorization boundary. The reviewed paper's stealth goal makes that distinction explicit: user-visible silence and internal state change can coexist.

Independent evaluation should therefore capture both policy decisions and raw state transitions. It should include true-benign legitimate updates, not only clean non-attacks, and should blind adjudicators to the method name. This is reviewer-derived methodology, not a result claimed by the paper.

### 10.3 Contemporaneous or later alternatives

Alternative defenses include signed source provenance, memory sandboxing, risk-scored quarantine, contradiction checks, retrieval-time labels, deterministic capability brokers, and independent confirmation. No single mechanism handles every stage. Sender authentication does not establish content truth; a classifier may miss adaptive text; quarantine may hurt personalization; user confirmation may create fatigue; and action brokers require complete mediation.

The best comparison is therefore a layered factorial: source authentication, memory authority, visible receipt, retrieval policy, and action authorization, tested alone and in combinations against paired benign/adversarial cases. The paper's decomposed metrics are well suited to this future study.

### 10.4 Publication, code, and reproducibility status

The reviewed object is arXiv:2607.05189v1, submitted 2026-07-06.[^source-paper] The paper states that the benchmark and dataset will be released, but the canonical record and manuscript did not publicly link code, data, checkpoints, or an experiment repository as of 2026-07-21. This is a time-bounded observation.

OpenClaw, Hermes Agent, and NanoClaw have public official repositories that provide runtime context.[^openclaw][^hermes][^nanoclaw] Those current repositories are not proof of the exact versions, defaults, or patches used in the paper. No repository code was executed in this review.

## 11. Research notes and critical considerations

### 11.1 Internal consistency

Three textual issues require correction in a future version. First, ESR is described both conditionally and as an all-run metric; the active equations and table relationships support all-run values. Second, the arXiv comment says eight figures, while the inspected paper contains ten numbered figures. Third, the limitations source includes an unfinished sentence ending in `xxx`. An RQ3 sentence also has a grammatical defect.

The complete source otherwise supports the major structure reported by the DEP-E: Sections I-VIII, Appendices A-G, Tables I-VII, Figures 1-10, optimization equations, ethics statement, and AI-use disclosure. The paper reports isolated Docker/simulated mail infrastructure, mock profiles, no real users, and planned responsible disclosure.

### 11.2 Metric and baseline concerns

The denominator ambiguity affects interpretation of ESR and derived comparisons. Every result release should include raw binary stage outcomes. The 20-case query-baseline subset prevents direct comparison with 56-case point estimates unless paired shared-case results are shown. Multiple model endpoints are compared without uncertainty, so differences of one or two cases should not become rankings.

Stealth rubrics also deserve user-centered validation. A model judge or predefined rubric can determine that a response is non-disclosing, but a real user may notice unusual wording, silence, or later behavior. Conversely, verbose receipts may be ignored. Security and usability need joint measurement.

### 11.3 Unsupported or overbroad claims

The experiments support a vulnerability class and transfer in tested configurations. They do not establish that all persistent agents are vulnerable, that a named vendor is intrinsically weaker, that defenses are generally ineffective, or that MemGhost will transfer after the architecture adds memory authority controls. The term "stealth" is bounded to evaluation rubrics, not covert behavior against a representative user population.

The absence of public artifacts also prevents the community from confirming exact method performance or benchmark quality. Reproducibility claims should remain prospective until the promised release is available and independently exercised.

### 11.4 Alternative explanations

Some reported transfer may arise because target agents share prompt patterns, tool conventions, or model-family behavior rather than because the learned policy captures a fully general memory-injection mechanism. Background advantage may reflect reduced response obligations rather than persistence itself. Defense failures may reflect threshold or integration choices. Adaptive visibility effects may arise from altered model instructions that also change writing behavior.

These explanations do not remove the vulnerability. They identify variables that a controlled factorial should isolate before causal claims are strengthened.

### 11.5 Unresolved questions

- How often do legitimate messages produce memory updates that resemble the attack under the same rubrics?
- Does a provenance-bearing `search_only` state preserve useful personalization while preventing sensitive actions?
- How do memories decay, merge, contradict, or recover over weeks of normal use?
- Can one compromised agent propagate a record into shared or multi-agent memory?
- Which result changes when sender authentication, enterprise filtering, and independent action approval are enabled together?
- How stable are proxy correlations after target models or system prompts change?

## 12. Potential implications

### 12.1 Scientific implications

Stateful-agent security needs longitudinal benchmarks. A single-turn answer is an incomplete unit of analysis when an observation can become state and later action. Evaluation should model temporal separation, state transitions, visibility, and downstream effects. WhisperBench is a useful step because it names those stages.

Attack optimization through proxies also creates a research problem in transfer measurement. Proxy validity should be treated like model calibration: measured across environments, monitored over time, and reported with failure regions. The high correlations in Table III motivate this work without resolving it.

### 12.2 System-design implications

Memory writes should be proposals, not implicit facts. A deterministic gateway should attach provenance and least authority. Retrieval should preserve labels. Prompt assembly should expose trust status to the model without relying on the model to enforce policy. Sensitive tools should require evidence and independent approval. Users need concise diffs and revocation controls.

The incident record should link source message hash, parsed content, extractor version, proposed and committed diffs, approval, retrieval, prompt assembly, plan, tool call, and outcome. Without that chain, a delayed failure is difficult to diagnose and remediate.

### 12.3 Broader or cross-domain implications

The same failure pattern can occur when agents ingest web pages, documents, tickets, code comments, meeting notes, CRM records, or other agents' messages. Any channel that becomes persistent context can import untrusted policy. The correct abstraction is not "email filter" but "state transition with authority."

Organizations also need ownership rules. A shared memory may contain data from several users and sources. Scope and tenancy must be enforceable properties, not natural-language conventions. Revocation should propagate to caches and derived summaries.

## 13. New hypotheses derived from the paper

These are reviewer inferences, not established findings.

### Hypothesis 1: Authority separation dominates content detection

**Proposition:** assigning external-message memories `search_only` authority and enforcing action-time evidence will reduce harmful downstream execution more reliably than improving a message classifier alone.

**Motivating evidence:** the paper reports high false-negative rates for tested defenses, while the attack requires retained state to influence later action.

**Predicted observation:** ISR may remain nonzero, but ESR and E2E for sensitive actions approach zero when every action path is completely mediated.

**Falsifying observation:** sensitive actions still execute from quarantined state despite complete broker mediation, or benign utility collapses beyond an agreed threshold.
**Minimum test:** a paired 56-case reproduction with content detection alone, authority separation alone, and both, plus legitimate-memory controls.

### Hypothesis 2: Signed memory diffs reduce stealth more than verbose narration

**Proposition:** a structured, deterministic receipt showing exact durable changes will improve detection and revocation more than asking the language model to explain what it remembered.

**Motivating evidence:** foreground visibility changes SSR/E2E sharply, and free-form responses are themselves controlled by the victim model.

**Predicted observation:** users identify unauthorized writes more accurately and faster with signed diffs, while receipt length remains bounded.

**Falsifying observation:** structured receipts do not improve recognition or create enough alert fatigue to erase the benefit.
**Minimum test:** a blinded user study comparing no receipt, model narration, and deterministic diff receipt on benign and unauthorized updates.

### Hypothesis 3: Proxy transfer decays after trust-boundary changes

**Proposition:** MemGhost's proxy correlations will degrade more after adding provenance and action authority than after changing only the victim language model.

**Motivating evidence:** current transfer spans several models and runtimes that still permit similar content-to-memory-to-action flow.

**Predicted observation:** model swaps reduce rates gradually, while a complete authority boundary creates a discontinuous drop in ESR/E2E.

**Falsifying observation:** one-shot generated content recovers high E2E without obtaining independent authority, indicating an unmediated path.
**Minimum test:** freeze the attacker policy and compare target model swaps against architectural policy changes with full lineage traces.

## 14. Deployment and governance considerations

### 14.1 Appropriate use

The paper and this review are appropriate for authorized defensive testing, memory-policy design, benchmark construction, incident-response planning, and evaluation of provenance-aware controls. Test accounts, synthetic identities, isolated mail infrastructure, non-production tools, and explicit authorization are required.

The work should inform threat models and control requirements. It should not be used to infer that a current named product is vulnerable without version-matched testing, nor to target real users.

### 14.2 Poor fit and failure modes

The benchmark is a poor substitute for production risk quantification. It omits organization-specific gateways, user behavior, access control, multi-tenancy, and long-horizon operations. A defense tuned only to WhisperBench may overfit its rubrics or content patterns. A receipt-only design may expose an unauthorized write without preventing it. A provenance-only design may authenticate a malicious sender without establishing content legitimacy.

Other failure modes include bypasses around the action broker, derived summaries that drop labels, caches that ignore revocation, ambiguous ownership of shared memory, and emergency paths that silently restore broad authority.

### 14.3 Required safeguards

1. Run evaluations only in isolated, authorized environments with synthetic accounts and no real-user data.
2. Withhold concrete payloads, triggers, and optimized attack corpora from public operational documentation.
3. Enforce complete mediation for sensitive tools and fail closed when provenance or approval is missing.
4. Preserve append-only lineage while minimizing personal data and applying retention limits.
5. Require user-visible, reversible memory changes and independent confirmation for authority elevation.
6. Publish model, runtime, evaluator, benchmark, and endpoint versions with uncertainty and clean-utility metrics.

## 15. Replication and falsification agenda

### 15.1 Highest-priority unresolved result

The highest-priority result is whether high one-shot background E2E survives a fully provenance-aware memory and action policy. Use the 56 held-out cases plus matched legitimate updates. Compare the paper-like baseline with a gateway that assigns external extractions `search_only`, a deterministic receipt, and an independent action broker. Success for the defense means no unauthorized sensitive action, complete lineage, and acceptable benign-update utility. A single unmediated sensitive action is a system-integrity failure and should produce a trace explaining the bypass.

### 15.2 Reproduce the full pipeline

Archive the exact 108 cases, split hashes, seed examples, attacker checkpoint, SFT and GRPO configurations, optimizer state, hard buffer, proxy versions, target runtime containers, endpoint dates, evaluator prompts, raw stage outcomes, and tool traces. Run at least five training seeds and repeated target evaluations where nondeterminism exists. Publish checksums and a machine-readable result schema.

The reproduction must remain defensive and access-controlled. Public results can include sanitized cases and aggregate traces; concrete optimized messages should be handled under a disclosure and misuse-review process.

### 15.3 Normalize evaluation

Evaluate every method on the same 20 shared cases and, where computationally feasible, all 56 held-out cases. Match target-query budgets or report a cost-success frontier. Include bootstrap confidence intervals and paired tests. Report both conditional and all-run ESR. Add benign messages, legitimate preferences, conflicting updates, expired instructions, and sender-authenticated but false content.

Record clean task completion, false-positive rate, confirmation burden, latency, token cost, storage cost, and revocation success. Security without utility and operator cost is not deployment evidence.

### 15.4 Test the proposed mechanism directly

Instrument every transition from mail receipt to memory write to retrieval to action. Randomize one control at a time: source label, authority class, receipt visibility, retrieval eligibility, and action approval. This determines whether visibility reduces stealth only, whether quarantine reduces execution, and whether independent confirmation prevents privilege elevation.

The primary mechanistic measure should be unauthorized authority acquisition: the proportion of external observations that become action-authorizing state without valid independent approval. This measure is closer to the system failure than semantic similarity or message classification alone.

### 15.5 Test realistic shifts and alternatives

Evaluate long-horizon memory decay and consolidation, repeated benign correspondence, multi-agent sharing, multi-user workspaces, multimodal attachments, multilingual mail, enterprise gateways, SPF/DKIM/DMARC outcomes, and compromised authenticated accounts. Freeze old attacker policies and measure transfer after model, runtime, prompt, memory, and policy changes separately.

Compare content screening, provenance, quarantine, user receipts, contradiction checks, retrieval gating, and action brokers in a factorial. Include adaptive attackers only under controlled review and avoid publishing operational payloads. Archive negative and null results.

## 16. Durable restatement

> A persistent agent is secure only if the path from observation to memory to action preserves trust. The reviewed paper reports that one untrusted email can create durable, low-visibility state and influence a later session across several controlled model and runtime configurations. The evidence establishes a serious vulnerability class, but not production prevalence or a universal bypass. The defensible response is to make memory writes explicit, provenance-bearing, least-authority, visible, reversible, and independently checked at action time.

The detail worth retaining is the separation of stages. Injection is not execution. Visibility is not prevention. Retrieval is not authorization. A strong architecture can store uncertain information for search or forensics while denying it authority over sensitive actions. A strong evaluation records every stage and reports the denominator, uncertainty, runtime coordinates, clean utility, and falsifier.
